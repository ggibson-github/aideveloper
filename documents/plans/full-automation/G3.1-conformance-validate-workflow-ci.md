## Behavior / step logic
<!-- timeline-source: agent cli-composer-2.5 2026-06-28 -->

1. On each pull request, GitHub Actions (or equivalent CI) runs `validate-workflow.py` and `headless-verify` as S0 gates—the same commands local pursuit uses from task cards and [F1.8](F1.8-pack-verify-goal-verify-suites.md) pack verify suites.
2. CI validates repo conformance and configured task evidence paths; it does not mutate `journal/state.json` or clear H1/H3 design gates in the pursuit state machine.
3. When a check fails, the workflow posts a comment linking to uploaded evidence logs so operators can triage without opening the IDE session.
4. Green CI is a prerequisite for merge and for downstream `goal_verify`, but passing checks alone does not mark a goal achieved or waive H3 sign-off.
5. If CI is misconfigured, unreachable, or returns indeterminate results, merge and goal-level verification fail closed at H2 until headless verify mirrors local `verify-router.py` behavior.

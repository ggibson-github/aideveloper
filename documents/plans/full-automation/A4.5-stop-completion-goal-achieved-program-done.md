## Reader narrative
<!-- prose-source: agent plane-a 2026-06-28 -->

Completion stops are successful termination: goal_verify passed, H3 accepted (or waived under policy), program manifest complete, or company_ops goal criteria satisfied per [INTRO-1.3](INTRO-1.3-goal-completion-criterion.md). The goal state transitions to achieved or program_done with timestamps and evidence pointers.

Unlike H3-pending ([A2.5](A2.5-goal-verify-pass-transition-h3-pending.md)), this stop closes pursuit for that goal id. Multi-goal company autopilot ([A3.3](A3.3-company-autopilot-multi-goal-role-workstreams.md)) may immediately advance to the next queued goal. Operators should see completion on dashboard/STATUS without parsing journals.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. When `goal_verify_command` passes and H3 accepts (or policy waives H3), pursuit records stop class `completion:achieved`—`goal.state` transitions to achieved with timestamps and evidence pointers per [INTRO-1.3](INTRO-1.3-goal-completion-criterion.md).
2. Unlike [A2.5](A2.5-goal-verify-pass-transition-h3-pending.md) H3 pending, completion closes pursuit for that goal_id—no further [A2.2](A2.2-if-ready-execute-one-pipeline-step.md) phases run against the same scope unless a new goal opens.
3. Program manifest complete or company_ops criteria satisfied produce analogous completion stops with roll-up fields for parent goals so dashboards show closure without journal parsing.
4. Under [A3.3](A3.3-company-autopilot-multi-goal-role-workstreams.md) company autopilot, completion on one goal may immediately advance scheduling to the next ready queued goal when preflight allows.
5. Premature completion without proof is blocked by [A4.2](A4.2-stop-verify-evidence-goal-verify-regression.md)—goal_verify must pass and H3 must clear before `completion:*` is written.

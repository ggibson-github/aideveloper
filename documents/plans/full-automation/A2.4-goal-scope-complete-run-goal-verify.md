# A2.4: goal_scope_complete run goal_verify

**Parent:** [A2-index](A2-index.md) · **Branch A** · **Vision §3** · **Release:** v2.14

## Reader narrative
<!-- prose-source: agent pilot 2026-06-28 -->

When all in-scope work for the goal is complete, pursuit shifts from step execution to goal-level proof. This capability defines the handoff: detect scope completion, then run goal_verify_command rather than another implement task.

Scope completion is machine-checkable—queue empty, task cards done, evidence present—not a conductor guess. Running goal_verify too early produces false failures; running it never leaves goals stuck in pursuing forever.

This node connects Plane A to Plane G. It is the pivot from “doing work” to “proving the goal.” Operators should expect a visible state transition to verifying before H3 is requested.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. When machine-checkable scope completion is true—task queue empty, task cards marked done, required evidence paths present—the conductor stops scheduling implement phases and pivots pursuit toward goal-level verification.
2. Preflight consults goal_scope_complete predicates derived from state.json and task rollups; the conductor does not invoke goal_verify on a guess or mid-stream partial completion.
3. On scope-complete signal, pursuit sets goal.state to verifying and runs goal_verify_command from [A1.3](A1.3-goal-verify-command-meta-test.md) instead of advancing next_action to another implement task.
4. Early goal_verify invocations while scope remains incomplete produce structured failures that keep goal.state in pursuing; deferred scope completion leaves the goal pursuing until predicates pass.
5. If scope signals conflict—queue shows empty but evidence is missing or H2 blockers remain—pursuit halts at H2 rather than entering verifying or claiming achievement without proof.

**Parent:** [B2-index](B2-index.md) · **Branch B** · **Vision §4** · **Release:** exists

## Reader narrative
<!-- prose-source: agent plane-b 2026-06-28 -->

Verifier and tool-operator are shell-class workers for bounded CLI runs: pytest via verify-router, Blender/Unreal export commands, checksum tools—anything with a literal command block on the task card. They write evidence logs under `evidence/` and set `last_verify` in state.

Non-pytest evidence types follow `docs/operator/evidence-types.md`. Verifier failure blocks task completion and may trigger [B3.3](B3.3-escalation-loop-on-verify-fail.md) escalation without clearing gates. Tool-operator never edits product source except via declared tool output paths.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. When a task card declares a Test command or Tool command, the conductor spawns Verifier or tool-operator shell workers to run the literal CLI block and capture output under `evidence/`.
2. verify-router.py dispatches pytest and records `last_verify` in state.json; non-pytest evidence follows `docs/operator/evidence-types.md` paths declared on the card.
3. Verifier and tool-operator workers do not edit product source except through declared tool output paths; they return bounded logs to the conductor for merge.
4. `last_verify=passed` is required before the conductor marks an evidence-required implement task complete or advances past implement-feature.
5. Verifier failure blocks task completion and may invoke [B3.3](B3.3-escalation-loop-on-verify-fail.md); workers cannot self-clear evidence gates without conductor re-run.

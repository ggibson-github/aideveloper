## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. On pack instantiation ([F2.1](F2.1-company-instantiate-program-scoper-pack-select.md)), S0 loads `roles.yaml` from the active template-pack and sets `state.active_role` to the pack default for the current workstream—each role encodes pipeline_id, enabled skills, and tool permissions per [B5.2](B5.2-role-to-pipeline-id-skills-tool-permissions.md).
2. Each pursuit wake reads `active_role` before [A2.1](A2.1-preflight-check-pipeline-blocked-extended.md) preflight so route-tier, spawn policy, and HITL strictness match pack intent—not the conductor's improvised defaults.
3. When switching roles mid-goal, the conductor records manifest approval or an explicit journal decision, updates `active_role` in the same dual-write turn, and re-runs preflight so workers receive refreshed spawn contracts with role-scoped `allowed_reads`.
4. Under [A3.3](A3.3-company-autopilot-multi-goal-role-workstreams.md) company autopilot, scheduling rotates across roles via [F2.3](F2.3-company-active-role-rotates-ready-work.md) without merging permissions—each spawned worker inherits only its role's pipeline_id and tool allowlist.
5. If `active_role` is missing, references an undefined pack role, or conflicts with manifest lane bindings, pursuit stops at H2 until operators repair pack config or record a valid role switch—never routing implement phases with ambiguous role context.

# B3.3: escalation loop on verify fail

**Parent:** [B3-index](B3-index.md) · **Branch B** · **Vision §4** · **Release:** exists

## Reader narrative
<!-- prose-source: agent plane-b 2026-06-28 -->

When verify fails, an escalation loop runs: capture log, classify failure (test, tool, environment, design), decide retry vs H2 vs refactor task—without silently marking the implement task complete. The conductor owns the loop; workers do not self-clear evidence gates.

Repeated verify failure on the same task card triggers S4 packaging for H2 with evidence paths attached. This prevents infinite implement→fail→implement churn without journal record.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. When [B2.4](B2.4-verifier-tool-operator-evidence.md) returns `last_verify=failed`, the conductor owns an escalation loop—capture the evidence log, classify failure as test, tool, environment, or design, and decide retry versus refactor task versus H2.
2. Workers do not self-clear evidence gates; only the conductor may re-spawn a Verifier after a bounded fix or record an H2 blocker with attached evidence paths.
3. The implement task stays incomplete and `next_action` does not advance past the failing task while `evidence_required` is true and verification has not passed.
4. Repeated verify failure on the same task card triggers S4 packaging per [B1.5](B1.5-s4-governance-escalation-h2-packaging.md) with evidence paths and journal notes so pursuit cannot churn implement→fail→implement silently.
5. If classification is ambiguous or recovery exceeds retry policy, pursuit halts at H2 with structured failure context rather than marking the task done or requesting H3 sign-off.

# B3.1: genius orchestration only thin turns

**Parent:** [B3-index](B3-index.md) · **Branch B** · **Vision §4** · **Release:** exists

## Reader narrative
<!-- prose-source: agent plane-b 2026-06-28 -->

Genius-tier models are for orchestration turns only: routing, merge, gate decisions, and spawn planning—not for writing hundreds of lines of feature code. `model-policy.json` maps tiers; route-tier.py applies capability_class S0–S4 to pick worker vs parent model.

Thin genius turns keep cost predictable and preserve worker parallelism. If the conductor implements large task cards inline while spawn_workers is true, that violates this capability and collapses tier economics.

## Purpose

B3.1 defines genius orchestration only thin turns for the agent-driven expert system. Cognition & routing — [S0](B1.1-s0-deterministic-mandatory-first.md)–S4, conductor, workers, dual-stack.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. [B1.1](B1.1-s0-deterministic-mandatory-first.md) route-tier and `model-policy.json` assign genius tier only for S3/S4 orchestration turns; implement-heavy work routes to economy workers when `spawn_workers` is true.
2. The conductor keeps each genius turn thin—spawn planning, merge, routing, and gate decisions—without implementing large task cards inline while `spawn_workers` remains true.
3. After S0 preflight, [B2.1](B2.1-conductor-genius-merge-route-platform-drain.md) reads `model_tier` from state.json and spawns workers rather than absorbing implement scope into the parent session.
4. Parallel spawn under orchestrate-subagents preserves worker budgets; the genius parent merges summaries via [A2.3](A2.3-post-step-route-tier-dual-write-increment.md) dual-write instead of re-implementing worker output.
5. If the conductor implements bulk code inline while `spawn_workers` is true, validate-workflow flags a tier violation and pursuit halts at H2 until routing resets to thin orchestration.

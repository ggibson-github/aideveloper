## Reader narrative
<!-- prose-source: agent plane-b 2026-06-28 -->

S0 is the capability class for behavior that already exists as a script under `scripts/`. Before any model improvises routing, validation, dashboard generation, or verify, the conductor must run the deterministic path and use its output.

This rule—mandatory first—prevents expensive re-derivation of state the repo already encodes. `route-tier.py`, `check-pipeline-blocked.py`, and `validate-workflow.py` are canonical S0 examples referenced throughout Plane A pursuit. When a new behavior stabilizes, promote it from S1 worker output to S0 script; do not leave file-derived logic in chat-only form. ## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. Before spawning workers or advancing `next_action`, the conductor runs applicable S0 scripts—`route-tier.py`, `check-pipeline-blocked.py`, and `validate-workflow.py`—and treats their stdout and exit codes as authoritative routing input.
2. Workers must not re-derive file-backed state in chat when an S0 script exists; the conductor passes script output into spawn contracts and journal dual-write per mandatory-first policy.
3. When implement work stabilizes repeatable file-derived behavior, promotion targets Plane D to extract an L2 script rather than leaving logic conductor-only in session memory.
4. S0 output that reports BLOCKED or conformance failure stops the turn before any S1–S4 worker runs, with journal and state.json recording the deterministic reason.
5. If script output and journal disagree, or a script is bypassed for convenience, pursuit halts at H2 until the conductor re-runs S0 preflight and reconciles dual-write.

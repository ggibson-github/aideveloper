# A3.1: session_autopilot max_steps_per_session

**Parent:** [A3-index](A3-index.md) · **Branch A** · **Vision §3** · **Release:** v2.15

## Reader narrative
<!-- prose-source: agent plane-a 2026-06-28 -->

Session autopilot is the in-IDE mode most operators use first: the conductor loops pursuit steps inside one Cursor session until a cap, a block, or goal completion. This capability defines `max_steps_per_session`—a safety valve so a single chat cannot run unbounded turns if preflight mis-reads state or a worker hangs.

The limit is not a product goal; goal autopilot (A3.2) and the SDK daemon (A3.4) lift it for long-running work. When the cap is hit, pursuit dual-writes a structured stop reason and exits cleanly so the next wake—manual Continue or daemon poll—resumes from the last good journal/state snapshot. See [A2.6 loop until blocked](A2.6-loop-until-blocked-budget-achieved-h3-reject.md) for the inner loop this mode wraps.

## Purpose

A3.1 defines session autopilot max steps per session for the agent-driven expert system. Pursuit & control plane — autonomous loops until goal verified.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. When autopilot.active is true in session_autopilot mode, the conductor wraps [A2.6](A2.6-loop-until-blocked-budget-achieved-h3-reject.md) inside one Cursor session—incrementing pursuit.session_step_count after each [A2.3](A2.3-post-step-route-tier-dual-write-increment.md) dual-write without waiting for manual Continue per [A2.7](A2.7-no-intermediate-wait-for-continue.md).
2. Each iteration runs [A2.1](A2.1-preflight-check-pipeline-blocked-extended.md) preflight then exactly one [A2.2](A2.2-if-ready-execute-one-pipeline-step.md) pipeline phase before post-step routing—a single skill phase per turn inside the session loop.
3. When pursuit.session_step_count reaches max_steps_per_session from state.json or operator policy, pursuit dual-writes stop reason session_cap and exits cleanly so the next wake—manual Continue or daemon poll—resumes from the last committed journal/state snapshot.
4. H2 blocks, H1/H3 touchpoints, [A2.4](A2.4-goal-scope-complete-run-goal-verify.md) scope completion, and [A1.4](A1.4-deadline-budget-steps-tokens-wall-clock.md) budget exhaustion terminate before the cap; [A3.2](A3.2-goal-autopilot-until-goal-verify-or-hard-block.md) goal autopilot lifts the session ceiling for long-running work.
5. If preflight mis-reads state or a worker hangs without incrementing counters, the session cap still forces a typed stop rather than unbounded turns in one chat—fail closed at H2 if dual-write of the cap stop fails.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. At H1 plan approval for a game-studio goal, program-scoper binds the active pack's `pipelines/*.yaml` so pursuit phases follow concept → mesh → rig → animation → engine import → QA → build in enforced order.
2. Each phase turn loads role-specific playbooks ([F1.6](F1.6-pack-playbooks-role-specific.md)) and seeded task cards ([F1.7](F1.7-pack-template-tasks-seed-cards.md)) so implement workers advance only the current pipeline gate—not downstream animator or build work.
3. The integration manifest declares which asset phases may parallelize; the conductor schedules lane workstreams without violating ordering gates (e.g. no animator tasks before rig evidence passes).
4. QA evidence must pass before build-phase tasks enqueue; partial phase completion rolls up in state but does not unlock ship until [F3.4](F3.4-game-studio-goal-verify-asset-engine-tests-perf.md) atop [F1.8](F1.8-pack-verify-goal-verify-suites.md) succeeds.
5. If a worker attempts an out-of-order phase, required prior gate evidence is missing, or parallel lanes collide on the same asset lease, pursuit halts at H2 until the conductor reorders work at H1.

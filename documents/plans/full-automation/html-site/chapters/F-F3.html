Example: game studio pack

F3: Example: game studio pack. Open each capability below for diagrams, verification, and contracts.

The game studio example pack defines roles: designer, technical artist, animator, programmer, QA, build engineer, release manager—each with pipeline slice, tools, and KPIs appropriate to concept-to-ship asset work. Roles map to F1.2 schema and demonstrate multi-disciplinary company pursuit under one pack_id.

This pack is reference, not mandatory—all studios need not use every role; program-scoper may defer lanes until milestone needs them. External tool bindings (F3.3) and pipelines (F3.2) specialize these roles. Cross-pack HR onboarding may import via F5.1.

Game studio pipelines walk concept → mesh → rig → animation → engine import → QA → build—each phase bound to roles and gates in pipelines/*.yaml. Phase order prevents animator work before rig exists and blocks build before QA evidence passes.

Pipeline phases reference pack playbooks (F1.6) and seed task cards (F1.7) for implement turns. Integration manifest declares which phases may parallelize (e.g. multiple asset lanes). goal_verify at ship uses F3.4 suite atop generic F1.8.

Game studio packs declare external tools: Blender, Unreal Engine, Perforce or Git, CI build farms—MCP servers, CLI allowlists, and tool-operator task card patterns per role (F6.3). Tool-operator and verifier workers run bounded exports and engine tests; they do not edit pursuit state.

Missing tool install or credentials is H2—not skip. Facts INDEX holds download URLs and credential locations (E1.4). Evidence for tool runs follows F6.4 output types (render checksum, export log, CI artifact path). Compose-first searches pack playbooks before improvising tool commands.

Game studio goal_verify proves assets load in engine, automated tests pass, and performance budgets hold—beyond per-task pytest evidence. The suite aggregates tool-operator logs, engine test output, and perf counters into one goal.verify_command resolved at scope complete (A2.4).

Weak goal_verify (smoke import only) is an accepted risk documented in adversarial review—pack authors tighten suites over releases. Failure blocks H3 and packages H2 with evidence paths attached. Extends generic pack verify (F1.8) without replacing task-level verify-router discipline.

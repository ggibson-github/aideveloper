D6.4 · Platform evolution plane

platform, done, staleness, node, and wired

platform done staleness node wired

Staleness graph wiring completes platform done: promoted artifact registers as a node in program or manifest staleness with upstream dependencies so reconcile-stale and reconcile-artifact-graph mark downstream design and tasks when the artifact changes (D5.2 extends bump this graph).

Missing staleness node means silent consumer drift after platform edit. Wire node id, dependency edges, and pack scope in manifest JSON. Together with D6.1–D6.3, D6.4 is the audit gate before queue item done—see D6-index.

How it works

  1. Step 1Before dequeue, platform done audit wires the promoted artifact as a staleness graph node in program or manifest JSON with upstream dependency edges and pack scope.
  2. Step 2D5.2 extend promotions bump this graph so reconcile-stale and reconcile-artifact-graph mark downstream design and tasks when the artifact changes.
  3. Step 3Together with D6.1–D6.3, D6.4 is the final audit gate. before the queue item marks done.
  4. Step 4Missing staleness node means silent consumer drift after platform edit—audit re-enqueues the promotion until node id and edges land on disk.
  5. Step 5If the conductor marks promotion done without staleness wiring, D6 audit fails closed at H2 and the head item stays on the platform queue.

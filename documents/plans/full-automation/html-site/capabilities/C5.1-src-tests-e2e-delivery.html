C5.1 · Product execution plane

src, tests, e2e, and delivery

src tests e2e delivery

Delivery layout standardizes src (or app root), tests/unit, tests/integration, tests/e2e when UI exists, docs paths from AGENTS.md. Scaffold and implement tasks must preserve layout so verify-router and CI find tests predictably.

Pack-defined layouts extend but do not break evidence paths without manifest update. Delivery structure is verify input for goal slices and pack conformance.

How it works

  1. Step 1After each completed product phase during autopilot, the conductor or S0 scheduler reads platform.drain_policy.product_steps_per_platform_turn (K, commonly 5) and evaluates whether steps_total % K == 0.
  2. Step 2When the modulo check is true and platform.promotion_queue is non-empty, the next pursuit iteration becomes a platform turn—D2.3 dequeues head instead of running implement, design, or another product next_action.
  3. Step 3When the promotion queue is empty at the K boundary, the scheduler skips the platform turn and product pursuit continues immediately—delivery does not idle waiting for self-improvement work.
  4. Step 4Pack operators tune K without forking schedulers via D5.1; baseline cadence still yields when D3.2, D3.4, or D3.5 overrides fire.
  5. Step 5If steps_total drifts from journal dual-write or K is misconfigured to zero, scheduling fails closed at H2 until state.json and drain policy are reconciled.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Edge case C5.1 variant 3: verify state dual-write before continuing pursuit.
  • Edge case C5.1 variant 4: verify state dual-write before continuing pursuit.

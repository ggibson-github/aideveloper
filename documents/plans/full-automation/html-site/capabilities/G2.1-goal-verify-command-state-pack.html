G2.1 · Verification & quality plane

goal verify command, state, and pack

goal_verify_command state pack

Goal-level verification begins with goal.verify_command resolved from state.goal or the active template-pack verify suite (F1.8). Pack authors declare the command once; consumer goals inherit it on instantiation rather than improvising per sprint.

The command is the contract for H3 readiness: unit, integration, e2e, and tool checks roll up under one invocable entry point (scripts/goal-verify.py or pack override). Missing or empty verify_command when goal scope is complete is an H2 blocker, not a silent skip.

How it works

  1. Step 1journal/progress.md mirrors machine state in prose—phase completions, Resolved Q&A, Open questions, Blockers, Evidence files, Context files, Session summary, and Next action—for operators who skim without parsing state.json.
  2. Step 2Only the conductor dual-writes the journal after each pipeline step via journal-keeper; workers return summaries but never edit progress.md directly.
  3. Step 3On continue/start, hooks may inject a journal summary (I1.4), but the conductor must still read the full file and any listed Context files before routing the next turn.
  4. Step 4Generated views such as STATUS.md and dashboard.md (I5.1) reflect journal content but are not authoritative—state.json remains the router.
  5. Step 5If journal Next action diverges from state.json next_action without a conductor dual-write in the same turn, pursuit stops at H2 until sync-state or manual reconciliation restores mirror integrity.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G2.1 variant 4: verify state dual-write before continuing pursuit.

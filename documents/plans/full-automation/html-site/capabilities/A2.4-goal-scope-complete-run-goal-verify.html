A2.4 · Pursuit & control plane

goal scope complete, run, and goal verify

goal_scope_complete run goal_verify

When all in-scope work for the goal is complete, pursuit shifts from step execution to goal-level proof. This capability defines the handoff: detect scope completion, then run goal_verify_command rather than another implement task.

Scope completion is machine-checkable—queue empty, task cards done, evidence present—not a conductor guess. Running goal_verify too early produces false failures; running it never leaves goals stuck in pursuing forever.

This node connects Plane A to Plane G. It is the pivot from “doing work” to “proving the goal.” Operators should expect a visible state transition to verifying before H3 is requested.

How it works

  1. Step 1When machine-checkable scope completion is true—task queue empty, task cards marked done, required evidence paths present—the conductor stops scheduling implement phases and pivots pursuit toward goal-level verification.
  2. Step 2Preflight consults goal_scope_complete predicates derived from state.json and task rollups; the conductor does not invoke goal_verify on a guess or mid-stream partial completion.
  3. Step 3On scope-complete signal, pursuit sets goal.state to verifying and runs goal_verify_command from A1.3 instead of advancing next_action to another implement task.
  4. Step 4Early goal_verify invocations while scope remains incomplete produce structured failures that keep goal.state in pursuing; deferred scope completion leaves the goal pursuing until predicates pass.
  5. Step 5If scope signals conflict—queue shows empty but evidence is missing or H2 blockers remain—pursuit halts at H2 rather than entering verifying or claiming achievement without proof.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A2.4 variant 4: verify state dual-write before continuing pursuit.

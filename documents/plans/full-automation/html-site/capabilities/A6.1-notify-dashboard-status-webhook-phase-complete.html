notify dashboard STATUS webhook phase complete

Notifications inform; they do not block pursuit. Phase completion—design artifact published, implement task verified, goal transition—updates docs/operator/dashboard.md, STATUS.md, and optional webhooks so operators have situational awareness without approving each step.

Regeneration is S0 where generate-dashboard.py exists. Webhooks fire on meaningful milestones, not every token, to avoid alert fatigue. This capability satisfies observability for unattended runs while preserving A6.3: reading STATUS never substitutes for H2 resolution.

How it works

  1. Step 1After each meaningful phase completion—design artifact written, implement task verified with evidence, or goal state transition—journal-keeper invokes S0 generate-dashboard.py to regenerate docs/operator/dashboard.md and STATUS.md without blocking the next pursuit turn.
  2. Step 2Configured webhooks fire on milestones (phase complete, H3 pending, goal achieved)—not every conductor turn—so A3.2 unattended runs remain observable without alert fatigue.
  3. Step 3Notifications dual-write snapshot fields in state.json for headless consumers (I5.1) while pursuit continues; notification failure never blocks A2.2 execution.
  4. Step 4Reading STATUS or dashboard output does not clear H2 blockers or substitute for operator action—A6.3 preserves strict HITL separation.
  5. Step 5If dashboard regeneration fails, the conductor logs the error and optionally enqueues a platform notification item—pursuit advances only when preflight is READY, not when the operator acknowledged the digest.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A6.1 variant 4: verify state dual-write before continuing pursuit.

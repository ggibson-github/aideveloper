rollback git branch per goal slice

Rollback at the git layer uses a branch per goal slice so failed pursue paths can revert without contaminating main or sibling goals. Git-workflow creates and tracks feature branches; goal completion merges only after goal verify and gates clear.

Operator-initiated rollback checks out the last known-good branch tip for that slice; conductor records branch name in journal handoff. Multi-goal company autopilot (A3.3) isolates slices to limit blast radius.

How it works

  1. Step 1At H1, when the operator supplies a spec or charter, the conductor anchors pursuit in state.json with north star, scope, and the minimal HITL contract (INTRO-1.2: H1 plan, H2 blocker, H3 sign-off).
  2. Step 2After H1 approval, goal autopilot drives pursuit until goal_verify passes or a hard stop from A4 fires—not when a chat session or token burst ends.
  3. Step 3On each wake, B1.1 runs before improvisation while the conductor executes one product pipeline step and peeks the platform promotion queue so scripts, playbooks, and checks discovered during delivery enqueue without pausing the active goal.
  4. Step 4When template-packs compose a company slice, roles, pipelines, and verification inherit catalog composition and the three structural shifts from this summary instead of reinventing setup per project.
  5. Step 5When scope or stop behavior is ambiguous, agents resolve conflicts using this executive summary before inventing new policy; unresolved contradictions escalate to H2 and fail closed.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G6.1 variant 4: verify state dual-write before continuing pursuit.

C5.2 · Product execution plane

environment, setup, tasks, ordered, and first

environment setup tasks ordered first

Environment setup tasks—portable Python, venv, VS solution, GitHub clone—must appear in task breakdown before library install tasks. AGENTS.md rule 17 requires the agent to install and verify local runnability, not leave setup to the user.

Ordering is enforced in task-breakdown skill behavior and audited in task card dependencies. Skipping env setup causes false implement complete when code cannot run locally.

How it works

  1. Step 1When D2.1 enqueue triggers fire, the conductor dual-writes one platform.promotion_queue[] row with required fields—id, source, target_level (L1–L5), priority, and effort_class—plus optional reason, fingerprint, and partial promotion state.
  2. Step 2On each pursuit wake, B1.1 schema validation runs before dequeue; malformed rows remain at queue head and pursuit fails closed at H2 until the conductor repairs or removes them.
  3. Step 3Schedulers read target_level and effort_class to route platform workers to the correct D1 ladder step and estimate drain cost before accepting a platform turn.
  4. Step 4When D3.2 or D3.5 fire, higher-priority items dequeue ahead of FIFO without altering sibling row shapes.
  5. Step 5Only the conductor may write queue JSON—concurrent manual edits reload each wake with a journal note; if a worker mutates platform.promotion_queue directly, scheduling and deduplication break and pursuit halts at H2 until state is reconciled.

decisions docs/decisions ADR

docs/decisions/ stores ADRs and recorded trade-offs when S3 ambiguity resolves: architecture choices, gate waivers with rationale, program manifest conflicts. Journal Q&A captures session decisions; ADRs persist cross-session authoritative rationale.

S3 turns in Plane B should end in journal entry or ADR (B1.4). ADRs link from design artifacts and task cards when implement depends on a settled choice. Unlike facts, decisions are repo-authored judgment—not external truth. Staleness may mark downstream design nodes when ADRs supersede prior assumptions (E5.1).

How it works

  1. Step 1Before git-workflow merge or program integration, the conductor runs S0 staleness scripts (update-staleness.py, reconcile-stale) and reads docs/manifest/staleness.json to confirm design artifacts match current pursuit scope.
  2. Step 2In program mode, reconcile-artifact-graph marks dependent manifest nodes stale when upstream HLD, DD, or integration manifest changed since last approved design.
  3. Step 3git-workflow blocks push and merge when the staleness graph lists unresolved stale nodes upstream of the implement diff—implement against outdated specs is the mistake class in G5.5.
  4. Step 4Economy workers may not waive merge on stale graph nodes; only the conductor may proceed after operator H2 approval or a completed reconcile-stale pass updates journal next_action.
  5. Step 5If staleness reconciliation is incomplete or scripts fail, pursuit fails closed at H2 rather than merging—preflight re-runs staleness checks on every wake after dual-write resume.

B3.1 · Cognition & routing plane

genius, orchestration, only, thin, and turns

genius orchestration only thin turns

Genius-tier models are for orchestration turns only: routing, merge, gate decisions, and spawn planning—not for writing hundreds of lines of feature code. model-policy.json maps tiers; route-tier.py applies capability_class S0–S4 to pick worker vs parent model.

Thin genius turns keep cost predictable and preserve worker parallelism. If the conductor implements large task cards inline while spawn_workers is true, that violates this capability and collapses tier economics.

How it works

  1. Step 1B1.1 route-tier and model-policy.json assign genius tier only for S3/S4 orchestration turns; implement-heavy work routes to economy workers when spawn_workers is true.
  2. Step 2The conductor keeps each genius turn thin—spawn planning, merge, routing, and gate decisions—without implementing large task cards inline while spawn_workers remains true.
  3. Step 3After S0 preflight, B2.1 reads model_tier from state.json and spawns workers rather than absorbing implement scope into the parent session.
  4. Step 4Parallel spawn under orchestrate-subagents preserves worker budgets; the genius parent merges summaries via A2.3 dual-write instead of re-implementing worker output.
  5. Step 5If the conductor implements bulk code inline while spawn_workers is true, validate-workflow flags a tier violation and pursuit halts at H2 until routing resets to thin orchestration.

runtime IDE genius conductor session

The in-IDE conductor session is genius-tier and thin: the operator selects the orchestration model once per session from docs/operator/model-policy.json, then the parent agent runs S0 preflight, spawns economy workers, and dual-writes journal/state—never bulk-implementing when spawn_workers is true.

This is the primary runtime for v2 pursuit: continue, autopilot, gate, and verify commands all assume a Cursor IDE session with AGENTS.md and rules loaded. See genius-conductor-tiered-routing.md and Vision §11 — Branch I — Runtime & integration plane.

How it works

  1. Step 1On continue/start, beforeSubmit and subagentStart hooks in .cursor/hooks.json run S0-thin scripts that inject journal summary, state.json snapshot fields, and context file paths into the conductor turn.
  2. Step 2preToolUse hooks enforce allowlisted commands and filesystem writes before economy workers execute shell or MCP tooling during implement tasks.
  3. Step 3preCompact fires sync-state snapshot writes paired with H6 so pursuit can resume after context compaction without losing the last good dual-write baseline.
  4. Step 4Hooks complement always-on rules from E4.1 but do not replace the continue skill requirement to read journal/progress.md each wake.
  5. Step 5If hook injection fails—corrupt state.json, missing journal, or script error—pursuit surfaces H2 and refuses the turn rather than running the conductor on stale or empty context.

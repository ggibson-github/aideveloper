A5.1 · Pursuit & control plane

continue, resume, pursuit, if, not, and blocked

continue resume pursuit if not blocked

“Continue” in the target architecture means resume the pursuit loop if preflight reports READY—not approval of design gates, not answers to open questions, and not permission to skip evidence. Manual /continue invokes one step when autopilot is off; daemons invoke the same semantics on poll.

This capability prevents the v2.13 failure mode where operators typed Continue dozens of times per feature. Blocked preflight must re-prompt for H2 assistance instead of advancing next_action. See A5.2 for what Continue explicitly does not mean.

How it works

  1. Step 1When the operator types /continue or a daemon poll fires with autopilot inactive, the conductor runs A2.1 via check-pipeline-blocked.py—advancing only when status is READY.
  2. Step 2If preflight reports BLOCKED from A4, Continue re-prompts for the specific human action instead of advancing next_action or treating the word as implicit approval.
  3. Step 3On READY, exactly one A2.2 skill phase executes followed by A2.3 dual-write—matching A2.7 one-step semantics.
  4. Step 4A3.4 SDK daemon invokes the same contract on poll when goal autopilot is off—IDE and headless runners share identical semantics.
  5. Step 5If the operator intended to approve a design gate or answer a blocker, pursuit stays blocked until an explicit approval phrase.—Continue alone never clears self-gates.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A5.1 variant 4: verify state dual-write before continuing pursuit.

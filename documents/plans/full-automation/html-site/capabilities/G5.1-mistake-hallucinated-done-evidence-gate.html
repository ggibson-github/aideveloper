G5.1 · Verification & quality plane

mistake, hallucinated, done, evidence, and gate

mistake hallucinated done evidence gate

Hallucinated done is declaring a task or goal complete without evidence files and passed verify—a common agent failure mode. Controls: evidence_required, mandatory verify-router runs, validate-workflow checks on last_verify, and conductor discipline never marking TODOS complete in prose alone.

Audit and hierarchy quality scripts re-enqueue deepen when queue items close without artifacts. This mistake class is the philosophical core of Plane G: success is demonstrated, not assumed.

How it works

  1. Step 1A leaf ID (e.g. G4.3, G5.2, F5.1), or
  2. Step 2The full prompt from the queue (includes reader narrative and output rules)
  3. Step 3After each verify-router run on a task card, S0 increments retry_count in state and compares it to the escalation threshold from the active pack.
  4. Step 4On the first repeated failure for the same task, the conductor spawns a bounded rework turn with the prior evidence log paths in allowed_reads—no silent autopilot retry loop.
  5. Step 5When retry_count crosses the S4 threshold, pursuit pauses at H2 with a structured escalation package: evidence paths, failure class, retry count, and suggested operator action.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G5.1 variant 4: verify state dual-write before continuing pursuit.

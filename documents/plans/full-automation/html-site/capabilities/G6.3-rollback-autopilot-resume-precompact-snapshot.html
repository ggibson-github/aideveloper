rollback autopilot resume preCompact snapshot

Autopilot resume after session compact or IDE restart reloads preCompact snapshot plus last good dual-write of journal/state.json—never chat memory. Hooks may inject a summary; conductor still reads full journal and Context files before continue.

Snapshot includes next_action, evidence_files, last_verify, gates_pending, and autopilot.active so check-pipeline-blocked reproduces the same stop/continue decision. Corrupt snapshot triggers validate-workflow failure and H2, not best-effort guess.

How it works

  1. Step 1On preCompact hook fire., sync-state writes a snapshot capturing next_action, evidence_files, last_verify, gates_pending, and autopilot.active before context compaction.
  2. Step 2After IDE restart or session compact, autopilot resume reloads the last good dual-write of journal/state.json plus the preCompact snapshot—never inferring pursuit state from chat memory.
  3. Step 3The conductor reads full journal/progress.md and Context files listed there before invoking continue, even when hooks inject a summary line.
  4. Step 4check-pipeline-blocked.py runs against restored state so stop/continue decisions match the pre-compact wake; corrupt snapshot fails validate-workflow and surfaces H2.
  5. Step 5If snapshot and journal/state disagree after reload, pursuit halts at H2 until the operator reconciles from evidence paths in H6 rather than best-effort guessing next_action.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G6.3 variant 4: verify state dual-write before continuing pursuit.

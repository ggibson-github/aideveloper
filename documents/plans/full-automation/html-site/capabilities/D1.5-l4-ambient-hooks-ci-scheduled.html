L4 ambient hooks CI scheduled

L4 is ambient automation: hooks in .cursor/hooks.json, CI workflows, and scheduled jobs that run without a conductor turn—staleness checks, dashboard regeneration, queue EMPTY gates, and validate-workflow on push. Ambient layers enforce invariants the pursuit loop might skip under token pressure.

Hook and CI extensions are platform work (D4.4) and must stay backwards-compatible or follow D5.2 staleness rules. L4 does not dequeue product next_action; it guards repo health while D3.1 interleaves explicit platform turns. See Vision §6 — Branch D — Platform evolution plane (parallel queue) for maturity ordering L0→L5.

How it works

  1. Step 1When a goal activates or switches company role, the conductor reads template-packs/*/README.md catalog nodes. compose-first—not pack internals by ad hoc path guess.
  2. Step 2Pack READMEs list roles, pipeline overrides, verify suites, and governance defaults as discoverable fragments that feed B5.2 role binding and the E1.7 platform umbrella.
  3. Step 3Librarian compose briefings rank pack README matches via E2.3 and set allowed_reads to the README plus bound fragment paths only for the pursuit turn.
  4. Step 4When vision or parent pack schema changes, E5.2 marks pack catalog nodes stale; reconcile-stale blocks instantiate or role-switch until README rows match current pack contents.
  5. Step 5If README lookup fails, active_role references a missing pack, or compose proceeds without catalog search, pursuit stops at H2 with structured dual-write—never bind improvised pack defaults.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Platform queue depth 0 but product blocked on missing playbook — D3.3 priority cut skips platform drain.
  • Edge case D1.5 variant 4: verify state dual-write before continuing pursuit.

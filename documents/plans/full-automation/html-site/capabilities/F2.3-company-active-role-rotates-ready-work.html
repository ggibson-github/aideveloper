F2.3 · Organization plane

company, active role, rotates, ready, and work

company active_role rotates ready work

Company autopilot rotates state.company.active_role based on ready work in lane queues—not fixed round-robin. When a role lane has unblocked tasks and leases available, conductor switches active_role, updates spawn contracts, and schedules pursuit for that persona (A3.3).

Rotation must release held lane leases before role switch (complete-work-order). Role switch updates tool permissions (F6.3) and allowed_reads (F6.2) without merging permissions across roles. Conductor identity persists across switches (F6.1).

How it works

  1. Step 1When company mode instantiates the data-platform template pack, B5.1 sets active_role to analyst, data engineer, DBA, SRE, or governance reviewer and binds each to pipeline stages defined in F4.2.
  2. Step 2Schema approval and regulated governance steps route to the governance reviewer role under pack strict_hitl policy—surfacing H2 blockers that A6.2 digests without stalling unrelated lanes.
  3. Step 3Production deploy and infrastructure commands execute only under the SRE active_role using tool permissions from F6.3; other roles fail closed if deploy verbs appear in their spawn contract.
  4. Step 4The genius-tier conductor rotates active_role across parallel workstreams. into each worker spawn.
  5. Step 5If active_role drifts from pack YAML bindings or a role lacks pipeline permissions for the current next_action, validate-workflow.py stops pursuit at H2 until the pack fragment reconciles.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Pack role handoff while lane lease held — complete-work-order releases lease before role switch.
  • Edge case F2.3 variant 4: verify state dual-write before continuing pursuit.

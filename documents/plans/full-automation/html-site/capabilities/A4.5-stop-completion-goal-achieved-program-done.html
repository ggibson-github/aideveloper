A4.5 · Pursuit & control plane

stop, completion, goal, achieved, program, and done

stop completion goal achieved program done

Completion stops are successful termination: goal_verify passed, H3 accepted (or waived under policy), program manifest complete, or company_ops goal criteria satisfied per INTRO-1.3. The goal state transitions to achieved or program_done with timestamps and evidence pointers.

Unlike H3-pending (A2.5), this stop closes pursuit for that goal id. Multi-goal company autopilot (A3.3) may immediately advance to the next queued goal. Operators should see completion on dashboard/STATUS without parsing journals.

How it works

  1. Step 1When goal_verify_command passes and H3 accepts (or policy waives H3), pursuit records stop class completion:achieved—goal.state transitions to achieved with timestamps and evidence pointers.
  2. Step 2Unlike A2.5 H3 pending, completion closes pursuit for that goal_id—no further A2.2 phases run against the same scope unless a new goal opens.
  3. Step 3Program manifest complete or company_ops criteria satisfied produce analogous completion stops with roll-up fields for parent goals so dashboards show closure without journal parsing.
  4. Step 4Under A3.3 company autopilot, completion on one goal may immediately advance scheduling to the next ready queued goal when preflight allows.
  5. Step 5Premature completion without proof is blocked by A4.2—goal_verify must pass and H3 must clear before completion:* is written.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A4.5 variant 4: verify state dual-write before continuing pursuit.

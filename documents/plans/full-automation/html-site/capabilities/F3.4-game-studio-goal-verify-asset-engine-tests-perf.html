F3.4 · Organization plane

game, studio, goal verify, asset, engine, tests, and perf

game studio goal_verify asset engine tests perf

Game studio goal_verify proves assets load in engine, automated tests pass, and performance budgets hold—beyond per-task pytest evidence. The suite aggregates tool-operator logs, engine test output, and perf counters into one goal.verify_command resolved at scope complete (A2.4).

Weak goal_verify (smoke import only) is an accepted risk documented in adversarial review—pack authors tighten suites over releases. Failure blocks H3 and packages H2 with evidence paths attached. Extends generic pack verify (F1.8) without replacing task-level verify-router discipline.

How it works

  1. Step 1When platform promotion lands playbooks, scripts, or pack fragments, reconcile-stale registers each reusable path in docs/manifest/staleness.json so downstream task cards and compose hits can mark dependencies stale.
  2. Step 2On script or playbook change during pursuit, the staleness graph propagates to dependent task cards and E2.2 compose results—workers reconcile before advancing implement steps.
  3. Step 3Regeneration of E1.1 and E1.7 reconciles graph entries with live catalog INDEX rows after platform drain completes.
  4. Step 4Pack verify suites from F1.8 may assert graph completeness at goal_verify—missing nodes fail closed rather than allowing pursuit on orphan artifact paths.
  5. Step 5If a promoted artifact lacks a staleness node or graph drift leaves dependents unmarked, pursuit stops at H2 until reconcile-stale or D4.5 restores traceability.

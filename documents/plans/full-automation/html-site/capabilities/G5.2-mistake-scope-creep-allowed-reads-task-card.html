G5.2 · Verification & quality plane

mistake, scope, creep, allowed reads, task, and card

mistake scope creep allowed_reads task card

Scope creep is editing paths outside allowed_reads or beyond the task card boundary—often from workers opening full design trees or “helpful” refactors. Controls: Librarian caps (B2.2), spawn contracts, and conductor merge that rejects out-of-scope diffs.

Task cards list explicit file targets; implement-feature workers must not expand. Violations trigger rework, not silent acceptance; repeated creep may escalate H2 with diff evidence.

How it works

  1. Step 1When the operator invokes /continue, /autopilot, /status, /gate, /task, /verify, /lane, or /program, the IDE loads the matching skill from .cursor/skills/ and runs S0 scripts before the conductor LLM turn.
  2. Step 2/continue executes exactly one pipeline phase.
  3. Step 3/verify routes to the verifier skill and verify-router.py; /lane and /program invoke orchestration skills—none of these commands clear gates_pending or substitute for H2 answers or H3 approval.
  4. Step 4Command stubs under .cursor/commands/ stay aligned with skill contracts so IDE sessions, I2.1 SDK runners, and headless entry points share the same pursuit semantics.
  5. Step 5If a slash command would advance past hitl.pending or design gates without explicit operator approval or waiver text in the journal, pursuit fail-closes at H2 rather than improvising.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G5.2 variant 4: verify state dual-write before continuing pursuit.

A5.3 · Pursuit & control plane

default after H1 auto-enter goal autopilot

default after H1 auto-enter goal_autopilot

After H1 approves a plan, the default transition is automatic entry into goal autopilot—no manual “start pursuit” step. The goal-keeper skill (or equivalent conductor path) sets autopilot.active, populates goal fields from A1, and runs the first preflight in the same session when possible.

This default encodes “100% automation between touchpoints”: the operator supplies intent once, then observes until H2 or H3. Opt-out remains for strict HITL packs or debugging single steps. Pair with A3.2 for the run mode entered here.

How it works

  1. Step 1When H1 plan approval clears hitl.pending and Resolved Q&A records the approval, goal-keeper sets autopilot.active and populates goal fields from A1 in the same dual-write turn—no separate manual start-pursuit command.
  2. Step 2The conductor immediately runs A2.1 preflight and, if READY, enters A3.2 goal autopilot in the same session when operator policy allows.
  3. Step 3This default encodes 100% automation between touchpoints: intent is supplied once at H1, then pursuit runs until H2, verify stop, resource stop, or H3.
  4. Step 4Strict HITL packs or debug single-step mode may opt out—autopilot.active stays false and A5.1 drives one phase per Continue instead.
  5. Step 5If goal fields are incomplete after H1, preflight fails closed at H2 rather than entering goal autopilot with ambiguous success_criteria or missing goal_verify_command.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A5.3 variant 4: verify state dual-write before continuing pursuit.

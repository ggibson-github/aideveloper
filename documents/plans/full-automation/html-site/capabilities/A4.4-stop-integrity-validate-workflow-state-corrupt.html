stop integrity validate-workflow state corrupt

Integrity stops fire when the harness cannot trust its own state: validate-workflow.py fails, state.json is corrupt or schema-invalid, dual-write mismatch between journal and state, or artifact graph references missing nodes. Pursuit must not continue on ambiguous memory.

Recovery is deterministic-first: repair scripts, restore snapshot from Plane H, or escalate H2 with the validation log attached. Continuing pursuit on corrupt state duplicates work or attaches evidence to the wrong goal. This stop class is rare in production but critical for restart safety after manual edits or merge conflicts.

How it works

  1. Step 1When preflight or post-step S0 runs validate-workflow.py against journal and state.json, any schema failure, conformance error, or dual-write mismatch triggers an A4.4 integrity stop before another A2.2 phase is scheduled.
  2. Step 2Pursuit halts immediately—the conductor does not spawn workers or advance next_action while harness memory is untrusted, so restart after manual edits or merge conflicts cannot duplicate or skip work on ambiguous state.
  3. Step 3Recovery is deterministic-first: run repair scripts, restore a Plane H snapshot, or reconcile journal and state.json until validate-workflow passes and A2.3 dual-write restores a single source of truth.
  4. Step 4If automated repair cannot restore valid state, preflight escalates H2 with the validation log attached and goal.state stays blocked until an operator approves recovery—never continuing pursuit on corrupt memory.
  5. Step 5On the next wake after recovery, A2.1 re-runs integrity checks fail-closed so evidence paths and pursuit counters cannot attach to the wrong goal_id.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A4.4 variant 4: verify state dual-write before continuing pursuit.

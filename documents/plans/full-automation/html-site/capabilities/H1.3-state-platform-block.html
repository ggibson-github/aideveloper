H1.3 · Persistence & state plane

state, platform, and block

state platform block

The platform block tracks promotion_queue head, drain counters, maturity ladders, and platform-turn scheduling separate from consumer product pursuit. Platform workers read this block; product workers must not mutate it except via conductor-orchestrated drain (B4.2).

Queue JSON lives under docs/automation/ with S0 peek/drain scripts; stale platform items defer with journal notes rather than silent drop. Platform idle drain when product waits on external H2 is D3.4. Dual-write updates platform counters alongside journal Platform section.

How it works

  1. Step 1Goal autopilot reads budget fields from A1.4—steps, tokens, wall-clock—deciding whether pursuit is unlimited or capped at daily checkpoints after H1 approval.
  2. Step 2Unlimited pursuit maximizes autonomy; checkpoint mode decrements budgets each turn and fires A4.3 resource stops when exhausted, with dashboard visibility per I5.1.
  3. Step 3Budget exhaustion is a hard stop in the A4 taxonomy—not a soft warning—so operators must explicitly raise caps or approve continuation at H2 before pursuit resumes.
  4. Step 4Until an ADR resolves unlimited versus checkpoint default, pursuit uses conservative daily step caps outside production while production requires operator opt-in via H1 plan fields.
  5. Step 5If budget counters drift from state.json, checkpoints are bypassed, or the dashboard cannot show remaining budget, pursuit fails closed at H2 rather than silently burning tokens.

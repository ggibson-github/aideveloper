APP-B · Front matter & cross-cutting architecture

state, json, and sketch

state json sketch

This appendix sketches additive state.json fields for v2 goals, platform queues, pursuit counters, HITL pending markers, and company context—without breaking state.json version 2 compatibility. Implementers use it as a map when reading Plane H specs.

The sketch is not a substitute for validate-workflow.py and schema tests; it orients readers before they open individual H1.x field specs.

flowchart TD
  trigger[Trigger condition] --> pre[Preconditions S0]
  pre --> exec[Execute step logic]

  verify -->|fail| h2[H2 blocker]

What this defines

  1. Step 1At each pursuit wake, S0 scripts and the conductor read legacy v2 router keys in journal/state.json first, then additive goal, platform, pursuit, hitl, and company blocks sketched here before any LLM interprets progress.
  2. Step 2New fields are additive only—sync-state.py and journal-keeper dual-write journal and state atomically so existing consumers of next_action, autopilot, and evidence_required keep working across restarts.
  3. Step 3Platform queue shape (H1.3), pursuit counters (H1.4), HITL pending markers (H1.5), and company context (H1.6) cross-reference Plane H field specs rather than duplicating them in this appendix.
  4. Step 4validate-workflow.py enforces schema conformance on read; corrupt or partial JSON triggers A4.4 stop before goal autopilot resumes.
  5. Step 5If a proposed state shape lacks schema tests or validate-workflow coverage, pursuit fails closed at H2—never write ambiguous fields that break resume from the last good dual-write.

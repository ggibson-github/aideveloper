runtime headless pull-ready complete-work-order lanes

Headless lane workers pull ready work orders from lane JSON when leases expire or slots open—complete-work-order scripts mark lane progress without IDE conductor presence (C3.2).

Pull-ready semantics require integration manifest approval and unblocked dependencies in artifact graph (H3). Headless lanes dual-write outcomes to journal/state via S0 scripts only; LLM workers return artifacts to queue for conductor merge.

How it works

  1. Step 1At H1 plan approval, the conductor stamps goal_id, optional parent_goal, and goal_type into state.json and journal—establishing the identity record every subsequent wake reloads before routing.
  2. Step 2On each pursuit wake, preflight reads goal_type to select contracts: milestone goals require manifest gates, feature goals bind task cards and evidence paths, and company_ops goals follow the ops queue.
  3. Step 3When parent_goal is set, child pursuits inherit roll-up scope so parallel program streams report status to the parent without duplicating identity fields at each leaf.
  4. Step 4The conductor and journal-keeper dual-write any H1 amendments to these fields; S0 scripts refuse progression if goal_id duplicates or parent_goal points to a missing ancestor.
  5. Step 5If identity drifts or conflicts—duplicate ids, wrong parent—pursuit halts at H2; evidence must not attach to the wrong scope, and goal verification cannot pass until fields are reconciled.

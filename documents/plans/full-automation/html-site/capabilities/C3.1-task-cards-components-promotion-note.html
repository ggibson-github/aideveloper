task cards Components Promotion note

Task cards are the atomic implement unit: scope, allowed_reads, files to touch, test command, evidence path, optional tool command. Components field links catalog entries; promotion_note captures platform queue candidates discovered during implement.

new-task-card.py generates consistent cards from templates. Cards must not duplicate full design docs—reference paths instead. Conductor assigns one card per implement turn when evidence_required is true.

How it works

  1. Step 1When next_action is implement-feature and evidence_required is true, the genius-tier conductor loads exactly one task card—scope, allowed_reads, files to touch, test command, evidence path, optional tool command—and passes bounded context to economy implement workers.
  2. Step 2The card's Components field records catalog entries composed via E2.4; workers must not expand beyond allowed_reads even when full design docs exist elsewhere in the repo.
  3. Step 3During implement, workers record promotion_note when they discover reusable scripts, playbooks, or skills that belong on the Plane D platform queue rather than duplicated inline in the feature.
  4. Step 4S0 new-task-card.py generates cards from templates at task-breakdown; the conductor references design paths in the card instead of embedding full HLD/DD prose so pursuit turns stay within token and permission bounds.
  5. Step 5If a card omits test or evidence commands, Components reference missing catalog nodes, or a worker marks done without verifier output, pursuit blocks at H2 and does not advance last_verify or close the implement task.

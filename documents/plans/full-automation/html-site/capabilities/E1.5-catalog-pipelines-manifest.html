E1.5 · Knowledge & composition plane

catalog, pipelines, and manifest

catalog pipelines manifest

The pipelines manifest under docs/manifest/pipelines/ declares pipeline_id values—greenfield, iterative_feature, program—with phase order, default skills, and verify suites. Instantiation sets pipeline_id in state.json; route-tier selects the active pipeline for next_action.

This catalog binds Plane E discovery to Plane C execution (C1.4). Pack-defined pipelines differ per company template without forking conductor code. Changes mark dependent task cards stale via E5.1 and may require reconcile-stale (E5.3).

How it works

  1. Step 1Before any shell or tool invocation, preToolUse hooks in .cursor/hooks.json evaluate the proposed command against role allowlists from B5.2 and active pack policy—deny is fail-closed with a journal note, not silent strip.
  2. Step 2Tool-operator and verifier workers run only bounded literal commands listed on the active task card; economy search and explore workers must not escalate to destructive or unapproved system commands.
  3. Step 3When a worker proposes a command outside the card or role permission set, the conductor rejects the invocation before merge rather than accepting out-of-allowlist shell access.
  4. Step 4Forbidden patterns—destructive file ops, credential exfiltration, disallowed network calls—block pursuit immediately at the hook layer without advancing implement or verify turns.
  5. Step 5If an unsafe command is attempted or a hook deny is overridden in chat, pursuit fails closed at H2 with command evidence attached until the operator approves a bounded alternative or updates pack policy at H1.

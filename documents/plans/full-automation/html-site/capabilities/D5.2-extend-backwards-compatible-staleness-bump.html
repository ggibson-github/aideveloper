extend backwards-compatible staleness bump

Extend means backwards-compatible improvement to an existing catalog artifact—add optional flag to script, append playbook section, extend skill without breaking callers. Extension triggers staleness bump on dependent design nodes and task cards per Plane E reconcile rules.

Platform queue items targeting extend must cite parent catalog id and compatibility proof—tests green, validate-workflow passes. Extend is preferred over fork when compose-first can absorb the change (B4.3). Non-compatible behavior change is fork, not extend, even if filename stays the same—semver and provenance matter.

How it works

  1. Step 1When a platform queue item targets extend, a worker improves an existing catalog artifact with backwards-compatible changes—optional script flags, appended playbook sections, extended skills—without breaking callers.
  2. Step 2The item must cite parent catalog id; the verifier runs tests and validate-workflow to prove compatibility before dequeue.
  3. Step 3On land, E5.1 bumps staleness on dependent design nodes and task cards per Plane E reconcile rules.
  4. Step 4Extend is preferred over fork when B4.3 compose-first can absorb the change; non-compatible behavior requires fork even if the filename stays the same.
  5. Step 5If compatibility proof fails or dependents remain stale without reconcile routing, pursuit stops at H2—never mark extend done while stale design artifacts block implement.

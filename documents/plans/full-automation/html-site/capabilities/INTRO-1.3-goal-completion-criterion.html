Goal completion criterion

A goal is achieved only when five conditions hold together: acceptance artifacts exist, automated goal verification passes, staleness and integration graphs are consistent, no blocking questions remain without deferred rationale, and platform debt for the goal is promoted or explicitly waived with expiry.

The agent does not stop for status theater. It stops on H1, H2, H3, unrecoverable failure, or resource budget caps. This criterion prevents the common failure mode of marking tasks done while the overarching goal remains unproven—Plane G goal_verify exists to enforce this bar.

flowchart TD
  trigger[Trigger condition] --> pre[Preconditions S0]
  pre --> exec[Execute step logic]

  verify -->|fail| h2[H2 blocker]

What this defines

  1. Step 1When A2.1 reports READY, the conductor reads next_action from state.json and invokes exactly one matching skill phase—implement-feature, dd-writer, or continue—never batching design, implement, and verify in a single wake.
  2. Step 2Each wake produces one ledger entry, one evidence slot, and one dual-write boundary so pursuit audit trails stay aligned with a single next_action transition.
  3. Step 3Economy workers run only when spawn_workers is true and B2.2 allowed_reads constrain scope; the conductor does not chain additional phases before post-step housekeeping.
  4. Step 4When the skill phase completes, control passes to A2.3 for route-tier, journal-keeper dual-write, and counter increment before the next preflight cycle.
  5. Step 5Violating one-step semantics—or executing when preflight was BLOCKED—halts pursuit at H2 because partial state makes resume ambiguous after crash, laptop sleep, or operator interrupt.

I4.1 · Runtime & integration plane

runtime external MCP browser DCC cloud

runtime external MCP browser DCC cloud

External MCP servers—browser automation, DCC tools, cloud APIs—extend worker capability per pack configuration. MCP tools appear in worker spawn contracts with explicit permission boundaries; the conductor approves which servers are active for active_role (B5.2).

Auth failures and rate limits surface H2 with credential pointers in facts INDEX. MCP is integration runtime, not pursuit router—state transitions remain conductor-owned. Browser MCP follows lock/navigate/unlock discipline to avoid rabbit-hole automation.

How it works

  1. Step 1When spawning workers for implement or tool-operator tasks, the conductor loads the active MCP server list from template-pack role permissions (B5.2) and injects only approved browser, DCC, or cloud API tools into the spawn contract.
  2. Step 2MCP tool calls execute inside the worker session as integration runtime—tool results return to the worker, but pursuit state transitions (next_action, goal.state, evidence_files) remain conductor-owned dual-writes to journal and state.json.
  3. Step 3Browser MCP sessions follow lock/navigate/unlock discipline: the worker acquires browser lock before navigation, completes the scoped action from the task card, then releases—preventing parallel rabbit-hole automation across pursuit turns.
  4. Step 4Auth failures, rate limits, or disallowed tool invocations abort the worker turn immediately; the conductor records structured H2 with credential pointers from docs/facts/INDEX rather than retrying with elevated permissions.
  5. Step 5DCC and cloud MCP outputs land only on task-card-declared paths; the verifier attaches logs to evidence/. before pursuit marks the implement step complete.

G5.5 · Verification & quality plane

mistake, stale, design, staleness, and reconcile

mistake stale design staleness reconcile

Stale design is implementing against outdated HLD, DD, or integration manifest while staleness graph marks dependents stale. Controls: reconcile-stale skill, reconcile-artifact-graph in program mode, and G4.4 merge gate.

Upstream vision or spec changes bump staleness; implement must pause or re-run design phases per journal next_action. Silent implement on stale nodes violates compose-first and expert-system integrity.

How it works

  1. Step 1After H1 approves a plan, pursuit binds pipeline_id and phase order from C1 in state.json; each wake executes exactly one phase skill (spec through implement). inside goal autopilot—not as a detached manual SDLC.
  2. Step 2Task cards from C3 scope economy worker turns; when evidence_required is true, verify-router must set last_verify: passed before next_action advances past the current implement task.
  3. Step 3Program mode fans parallel lanes from work orders and the integration manifest.; the conductor orchestrates ready lanes without clearing manifest or milestone gates without explicit H1 approval.
  4. Step 4Per-task evidence and definition-of-done from C5 roll up toward goal_verify.—passing individual tasks does not close the goal without Plane G aggregation.
  5. Step 5If a design gate is pending, task evidence is missing, or workers would edit out-of-scope paths, Plane C halts at H2 fail-closed rather than marking phases complete or skipping to implement.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G5.5 variant 4: verify state dual-write before continuing pursuit.

preflight check-pipeline-blocked extended

Every pursuit turn starts with a question: is it safe to act? This capability extends preflight via check-pipeline-blocked.py so the conductor never executes a pipeline step while gates, blockers, budgets, or missing evidence would make that step invalid.

Exit code semantics are strict: READY (0) means exactly one skill phase may run; BLOCKED (1) means stop, dual-write a structured H2 or stop reason, and do not advance next_action. Preflight is S0—deterministic, cheap, and mandatory before any model-heavy work (B1.1 S0 rule).

Skipping preflight is how autonomous systems waste tokens on work that cannot commit. This step is the first node in the A2 pursuit loop and re-runs after every completed turn.

flowchart TD
  start[Conductor turn start] --> s0[check-pipeline-blocked.py]
  s0 -->|exit 0 READY| step[Execute one pipeline step]
  s0 -->|exit 1 BLOCKED| stop[Stop pursuit / H2]
  step --> post[route-tier + dual-write]
  post --> start

How it works

  1. Step 1Every pursuit wake—session autopilot, goal_autopilot, or SDK daemon—starts with the conductor running python scripts/automation/check-pipeline-blocked.py (S0) against journal and state.json before any skill phase or worker spawn.
  2. Step 2Exit code READY (0) authorizes exactly one pipeline phase via A2.2; BLOCKED (1) halts the turn, dual-writes a structured stop reason, and leaves next_action unchanged.
  3. Step 3Preflight tests gates_pending, blocking_questions, A1.4 budget caps, evidence_required gaps, and A1.5 goal.state so model-heavy work never starts when the step cannot commit.
  4. Step 4Per B1.1, routing and worker spawns never precede this S0 check; A2.3 re-invokes preflight on the next wake after each completed phase.
  5. Step 5If BLOCKED fires for corrupt state, unresolved gates, or stale evidence, pursuit routes to H2 and goal.state becomes blocked until the operator or journal-keeper reconciles journal and state.json.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A2.1 variant 4: verify state dual-write before continuing pursuit.

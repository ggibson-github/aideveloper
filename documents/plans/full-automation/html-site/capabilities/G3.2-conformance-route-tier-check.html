conformance route-tier check

route-tier conformance ensures next_action, capability_class, spawn_workers, and model tier align whenever the conductor changes pursuit direction. python scripts/route-tier.py --apply is the canonical apply path; validate-workflow cross-checks tier output against state.json.

Mis-routed genius implement or missing spawn_workers when task cards expect workers is a conformance failure, not a soft warning. Pack role bindings (B5.2) feed expected tier patterns for validation.

How it works

  1. Step 1After H1 approves the plan, pursuit owns SDLC execution—implement, test, refactor, integrate, and deploy preparation—without requiring a human turn for each phase transition.
  2. Step 2In-scope work runs through A3.2: multi-role company workflows via template-packs, structured H2 blocker escalation, and platform-queue harness self-improvement alongside the active product goal.
  3. Step 3Out-of-scope work—irreversible production without verify and rollback, silent waiver of safety gates, subjective aesthetic approval unless encoded in acceptance tests—must not enter autonomous pursuit; pack authors classify workflows against this boundary before H1.
  4. Step 4When a step needs authority, judgment of record, or external access the system cannot obtain, preflight raises H2. instead of improvising past the three touchpoints.
  5. Step 5If scope ambiguity would let agents advance past verify or waive gates silently, check-pipeline-blocked.py stops pursuit fail-closed until operators clarify boundaries via journal and state.json dual-write.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G3.2 variant 4: verify state dual-write before continuing pursuit.

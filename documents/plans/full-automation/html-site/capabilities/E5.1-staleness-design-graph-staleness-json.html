staleness design graph staleness.json

docs/manifest/staleness.json is the design-graph staleness manifest: nodes for HLD, DD, diagrams, task cards with dependency edges and stale flags when upstream artifacts change. S0 update-staleness.py (or successor) maintains the graph file-derived.

Staleness blocks implement spawn against obsolete design until reconcile completes. Program artifact graphs (C4.4) complement this manifest for multi-stream work. Vision § changes enqueue deepen items for affected hierarchy ids. See E5-index for sibling extensions.

How it works

  1. Step 1After design artifacts change (HLD, DD, diagrams, task cards), S0 update-staleness.py derives dependency edges and sets stale flags in docs/manifest/staleness.json without conductor improvisation.
  2. Step 2On each pursuit wake before implement spawn, preflight reads staleness.json; paths touching stale design nodes block worker spawn until reconcile-stale clears flags or records an H2 waiver.
  3. Step 3In program mode, artifact graphs from C4.4 complement the design manifest so cross-stream contracts surface stale nodes the global manifest alone might miss.
  4. Step 4When vision or hierarchy source changes, affected node ids enqueue deepen items so the staleness graph and pursuit queue stay aligned with approved scope.
  5. Step 5If staleness.json is missing, corrupt, or shows unresolved stale nodes on a path about to implement, pursuit stops at H2—silent implement against obsolete design violates Plane E and undermines goal_verify evidence.

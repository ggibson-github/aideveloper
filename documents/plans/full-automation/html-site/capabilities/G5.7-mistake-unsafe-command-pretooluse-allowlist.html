mistake unsafe command preToolUse allowlist

Unsafe commands are shell invocations outside role allowlists—destructive rm, credential exfiltration patterns, or network calls forbidden by pack policy. Controls: preToolUse hooks in .cursor/hooks.json, per-role tool permissions from template-pack (B5.2), and conductor rejection of worker-proposed commands not on the task card.

Tool-operator and verifier workers run bounded literal commands only; economy search must not escalate to unapproved system commands. Hook deny is fail-closed with journal note, not silent strip.

How it works

  1. Step 1When strict_hitl is off and pursuit reaches HLD, DD, or manifest gates, the conductor reads the configured self-gate mode from state—checklist-only, automated LLM reviewer, or second economy reviewer—before treating a middle gate as cleared without H3.
  2. Step 2In checklist-only mode, gate clearance requires every checklist row to map to an evidence file or verify command that G2.1 can audit; the conductor must not advance next_action on conductor prose alone.
  3. Step 3When reviewer mode is active, a genius-tier B2.5 subagent evaluates the artifact under scoped allowed_reads and records pass/fail in state before design phases resume.
  4. Step 4Until an ADR resolves this open decision, pursuit defaults to checklist plus machine-checkable evidence paths so middle gates stay fail-closed under A5.2.
  5. Step 5If self-gate mode is undefined, reviewer output conflicts with evidence, or checklist items lack proof, pursuit stops at H2—middle design gates never waive silently.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G5.7 variant 4: verify state dual-write before continuing pursuit.

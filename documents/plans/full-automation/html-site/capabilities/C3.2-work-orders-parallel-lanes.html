C3.2 · Product execution plane

work, orders, parallel, and lanes

work orders parallel lanes

Work orders group task cards for parallel lanes under program mode: lane json, leases, and manifest dependencies prevent two workers editing the same journal writer or conflicting branches.

Orchestrate-program spawns only when manifest gate cleared and spawn_policy allows. Single-stream greenfield uses sequential cards without work orders unless user enables program features.

How it works

  1. Step 1When any child catalog INDEX changes—scripts, playbooks, skills, facts, pipelines, or pack README entries—B1.1 S0 regeneration rebuilds docs/platform/CATALOG.md so the umbrella stays the single merged discovery surface before the next pursuit wake.
  2. Step 2At compose-first preflight. list-components runs against CATALOG.md and child INDEX manifests to return candidate paths, maturity tier, and component type.
  3. Step 3The umbrella satisfies E1's single discovery surface goal: pack README fragments from E1.6 appear alongside platform catalogs so role binding and compose ranking share one entry point.
  4. Step 4Bidirectional cross-links to child INDEX files must hold; when E5.1 marks catalog nodes stale, platform maturity claims in Plane D block promotion-done until umbrella completeness and staleness graph coverage reconcile.
  5. Step 5If CATALOG.md is missing, out of date with child INDEX rows, or compose returns empty without E2.5 promotion enqueue, pursuit stops at H2—never improvise catalog paths from conductor chat.

I2.2 · Runtime & integration plane

runtime SDK CURSOR API KEY AUTOPILOT MODEL

runtime SDK CURSOR_API_KEY AUTOPILOT_MODEL

SDK runtime requires CURSOR_API_KEY and optional AUTOPILOT_MODEL override for headless or remote worker sessions. Secrets never commit to repo; operators configure via environment or local secrets store documented in facts INDEX.

Model override must still conform to model-policy.json tier mapping—daemon does not bypass genius/economy routing for conductor vs worker roles. Missing credentials produce structured H2 with setup instructions, not silent fallback to unauthenticated calls.

How it works

  1. Step 1Before I2.1 invokes Cursor CLI or cursor-sdk for a headless pursuit turn, preflight loads CURSOR_API_KEY from environment or the operator secrets store listed in docs/facts/INDEX—never from committed repo files.
  2. Step 2Optional AUTOPILOT_MODEL overrides the default slug for autopilot batches, but J1 and model-policy.json still bind conductor roles to genius tier and worker roles to economy—override cannot bypass subagent_models routing.
  3. Step 3Each SDK daemon wake passes the resolved API key and model selection into Agent.prompt or headless CLI (agent -p); authentication failures abort the turn immediately with structured H2 citing credential setup steps.
  4. Step 4Credential rotation mid-pursuit requires the operator to update environment or the secrets store; the next daemon wake reloads config, dual-writes the change to journal and state.json, and resumes only after preflight re-authenticates.
  5. Step 5Missing or invalid credentials never fall back to unauthenticated API calls—pursuit stops fail-closed with H2 and may notify via I5.2 rather than silently retrying.

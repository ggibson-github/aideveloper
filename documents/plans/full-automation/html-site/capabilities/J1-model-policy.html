J1 · Governance & operator plane

model and policy

model policy

docs/operator/model-policy.json defines genius and economy tier Cursor model lists, autopilot defaults, spawn_workers policy, and capability_class routing hints. Operators update tiers as models evolve—fixed model names are not hardcoded in skills or rules.

route-tier.py reads policy plus state to set model_tier each turn (B3.1). SDK and IDE runtimes share the same policy file (I2.2). Policy changes should run validate-workflow.py before unattended autopilot resumes. See Vision §12 — Branch J — Governance & operator plane.

How it works

  1. Step 1On each pursuit wake, the conductor runs scripts/route-tier.py --apply (S0) against docs/operator/model-policy.json and journal/state.json to set model_tier, spawn_workers, and subagent_models before any skill phase runs.
  2. Step 2Genius-tier models from policy bind to the conductor orchestration turns under B3.1; economy tiers route to spawned workers per capability_class—fixed model names never hardcode in skills or rules.
  3. Step 3IDE sessions and SDK daemons (I2.2) load the same policy file so autopilot routing stays consistent across runtimes.
  4. Step 4When operators edit tier lists or spawn defaults, validate-workflow.py must pass before unattended goal autopilot resumes so corrupt policy cannot silently misroute turns.
  5. Step 5If policy JSON is missing, invalid, or contradicts state capability_class, preflight fails closed at H2 rather than proceeding with an undefined model tier.

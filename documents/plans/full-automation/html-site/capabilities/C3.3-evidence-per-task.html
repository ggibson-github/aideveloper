evidence per task

Evidence per task is mandatory when evidence_required: verify-router or tool-operator writes logs under evidence/, state records evidence_files and last_verify. Implement tasks do not complete on agent assertion alone.

Evidence types include pytest logs, tool output, checksums—see docs/operator/evidence-types.md. Missing evidence blocks goal rollup in C3.4 and goal_verify in Plane A/G.

How it works

  1. Step 1During implement or verify turns, economy workers detect repeated file patterns, test fixes, or catalog misses across pursuit iterations and emit a promotion flag in their return contract—the conductor receives the signal; workers never mutate platform.promotion_queue directly.
  2. Step 2The genius-tier conductor merges worker flags after each pursuit step, deduplicates by content fingerprint, and enqueues one promotion_queue item per confirmed pattern at the appropriate target_level from the D1 promotion ladder.
  3. Step 3Structural rework invisible to shell history—duplicate scaffolding, recurring verify failures on the same fixture, or compose-first misses—routes here; pair flags with B4.4 divergence logs when catalog composition failed before invention.
  4. Step 4A single worker flag defers enqueue until a second independent flag or the D2.1.1 manual 2× command trigger confirms the pattern, avoiding false-positive platform work.
  5. Step 5If workers write queue JSON directly, deduplication fails and duplicate promotion items stall the parallel queue, pursuit fails closed at H2 until the conductor reconciles flags and dual-writes a single enqueue record per pattern.

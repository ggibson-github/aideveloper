APP-A-verify · Front matter & cross-cutting architecture

Work taxonomy verify

verify work taxonomy verify

Verify work spans unit, integration, E2E, tool outputs, goal_verify, and conformance scripts. Verification is not a phase humans optionally trigger—it is the gate that separates pursuing from achieved.

Pack workflows must name verify commands explicitly so goal_verify can aggregate them.

flowchart TD
  trigger[Trigger condition] --> pre[Preconditions S0]
  pre --> exec[Execute step logic]
  exec --> verify[Verify output]
  verify -->|pass| done[Mark queue item done]
  verify -->|fail| h2[H2 blocker]

What this defines

  1. Step 1Pack workflows bind each verify slice to explicit unit, integration, E2E, tool-operator, or conformance commands so G2.1 goal_verify can aggregate them mechanically—never subjective "looks good" gates.
  2. Step 2When evidence_required is true, the conductor spawns Verifier or tool-operator workers via G1.1 and verify-router.py with the task card's literal Test or Tool command block rather than improvising checks.
  3. Step 3Shell workers write logs under evidence/ per docs/operator/evidence-types.md and return paths; the conductor dual-writes evidence_files and sets last_verify to passed before advancing next_action or marking implement tasks complete.
  4. Step 4Conformance scripts (G3.1, route-tier checks) run at pursuit boundaries so goal.state cannot move from pursuing to achieved without aggregated proof across named commands.
  5. Step 5On verify failure, missing evidence, or goal_verify aggregation gaps, pursuit fails closed at H2—G1.3 blocks advance and B3.3 may escalate without clearing gates.

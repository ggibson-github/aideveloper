A1.5 · Pursuit & control plane

Goal state enum pursuing blocked verifying achieved rejected

goal_state enum pursuing blocked verifying achieved rejected

Operators and dashboards need a single, honest lifecycle state for each goal. This capability defines the goal state enum: pursuing, blocked, verifying, achieved, and rejected—transitions driven by preflight, step execution, goal_verify, and H3 outcomes.

The enum prevents ambiguous records such as “done” without verification or “in progress” while H3 is pending. Blocked correlates with H2 assistance; verifying means scope is complete and goal_verify is running; rejected captures H3 denial with notes so pursuit can resume with corrected criteria.

State must stay synchronized across journal/progress.md and state.json. Resume after laptop sleep or daemon restart relies on this enum being authoritative. See the A2 pursuit loop for when each transition fires in sequence.

How it works

  1. Step 1At H1 plan approval, the conductor sets goal.state to pursuing in state.json—dual-written to journal—alongside A1.1 identity fields so every wake and dashboard reads the same lifecycle starting point.
  2. Step 2On each A2.1 wake, READY keeps goal.state at pursuing while BLOCKED sets blocked and routes to H2 assistance; preflight never schedules pipeline phases when the enum is blocked or verifying.
  3. Step 3When A2.4 marks scope complete, the conductor transitions goal.state to verifying and runs A1.3 goal_verify before requesting operator sign-off.
  4. Step 4Passing goal_verify advances to A2.5 H3 pending without claiming achieved; H3 acceptance sets achieved while H3 rejection records structured notes and returns goal.state to pursuing with corrected criteria.
  5. Step 5If journal and state.json disagree on goal.state after laptop resume or concurrent edit, preflight fails closed at H2 until journal-keeper reconciles the enum before pursuit continues.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A1.5 variant 4: verify state dual-write before continuing pursuit.

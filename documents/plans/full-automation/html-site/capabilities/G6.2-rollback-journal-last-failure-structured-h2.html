rollback journal last_failure structured H2

Structured last_failure in journal and state captures verify regressions, conformance failures, and H2 blockers with goal id, evidence paths, and suggested operator action—enabling resume without re-deriving context. Rollback decisions read this block before rewinding git or resetting next_action.

Empty or vague failure notes violate S4 packaging (B1.5). Dual-write failure metadata in the same turn as stop so G6.3 snapshots remain trustworthy.

How it works

  1. Step 1Conductor appends spawn metadata to worker-runs.jsonl on each orchestrate-subagents spawn; workers never touch journal/state.
  2. Step 2Outcome summary and duration logged on return for J4 governance.
  3. Step 3Parallel lanes correlated by goal_id + C4.1 lease.
  4. Step 4Failures/timeouts logged before H2; scope bleed visible when allowed_reads exceeded.
  5. Step 5Retention per operator pack; default full local history for autopilot resume.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G6.2 variant 4: verify state dual-write before continuing pursuit.

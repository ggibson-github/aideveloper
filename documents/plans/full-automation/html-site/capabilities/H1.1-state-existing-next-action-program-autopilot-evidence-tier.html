H1.1 · Persistence & state plane

State existing next action program autopilot evidence tier

state existing next_action program autopilot evidence tier

The v2 machine router in journal/state.json carries legacy pursuit fields—next_action, program, autopilot, evidence_required, model_tier, spawn_workers—that route-tier.py and validate-workflow.py read every turn. These top-level keys remain the conductor's first S0 lookup before any LLM interprets progress.

Additive v2 blocks (goal, platform, pursuit, hitl, company) extend without breaking existing consumers; sync-state.py and journal-keeper dual-write both files atomically where possible. Corrupt or partial state triggers A4.4 stop. See APP-B-state-json-sketch and Vision §10 — Branch H — Persistence & state plane.

How it works

  1. Step 1On every pursuit wake, S0 reads top-level next_action, program, autopilot, evidence_required, model_tier, and spawn_workers from journal/state.json before the conductor interprets journal/progress.md (B1.1).
  2. Step 2route-tier.py --apply and validate-workflow.py consume these legacy router fields to select pipeline phase, model tier, and worker spawn policy without re-deriving state in the LLM.
  3. Step 3Additive v2 blocks (goal, platform, pursuit, hitl, company) extend the router atomically via journal-keeper and sync-state.py while preserving existing consumer keys.
  4. Step 4When autopilot.active is true, next_action advances only through S0 apply paths—manual journal edits without dual-write do not change the machine router.
  5. Step 5Corrupt or partial state.json triggers A4.4 integrity stop with H2; pursuit does not resume until validate-workflow passes.

G4.4 · Verification & quality plane

review, staleness, reconciliation, before, and merge

review staleness reconciliation before merge

Before merge or program integration, staleness reconciliation must show design artifacts current: docs/manifest/staleness.json and reconcile-stale/reconcile-artifact-graph outputs gate git-workflow when upstream HLD, DD, or manifest nodes changed.

Review at merge time catches implement against stale specs—a mistake class mapped in G5.5. Conductor runs S0 staleness scripts; workers do not waive merge on stale graph nodes without operator approval.

How it works

  1. Step 1On each pursuit wake, route-tier.py reads capability_class (S0–S4) from state.json and assigns genius conductor versus economy worker roles per B1 before any implement turn runs.
  2. Step 2The genius-tier parent orchestrates thin turns only—spawning economy workers for bounded task cards per B2.1 and B3.1 when spawn_workers is true.
  3. Step 3B1.1 runs mandatory scripts/ output before improvisation whenever file-derived behavior exists; Plane B enforces this ahead of LLM routing on every wake.
  4. Step 4Model tier selection from docs/operator/model-policy.json maps conductor and worker models to pursuit cost and risk class per B3, while B4 interleaves product next_action steps with platform promotion queue peeks.
  5. Step 5If routing would assign S4 governance work to economy workers or skip S0 when a script exists, pursuit stops at H2 with a capability-class violation until the conductor realigns with model policy.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G4.4 variant 4: verify state dual-write before continuing pursuit.

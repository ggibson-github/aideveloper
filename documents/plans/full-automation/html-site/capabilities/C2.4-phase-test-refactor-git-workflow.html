phase test refactor git-workflow

Test and git-workflow phase covers test-writer, full suite runs before push, refactor when debt blocks features, and PR/commit when user requests. test-before-push rule applies: unit, integration, e2e when UI exists.

git-workflow runs only when last_verify passed or exceptions documented. This phase connects Plane C delivery to operator git policy without bypassing evidence gates.

How it works

  1. Step 1During task-breakdown after DD approval, the conductor sequences environment setup tasks—portable Python, venv, Visual Studio solution, GitHub clone—before any library or package install tasks in TODOS and task card dependencies.
  2. Step 2Implement-feature workers execute env-setup cards first; pip, NuGet, or package installs run only inside or against the runtime established by those completed setup tasks per AGENTS.md rule 17.
  3. Step 3Task card dependency edges and audit scripts verify ordering—an implement card cannot advance while its prerequisite env-setup card lacks last_verify=passed and evidence on disk.
  4. Step 4When scaffold or pack C1.4 introduces new runtime requirements, task-breakdown inserts additional env-setup rows at the front of the breakdown before downstream implement turns route from state.json.
  5. Step 5If library install tasks run before env setup completes, local runnability checks fail, or task dependencies skip setup ordering, pursuit fails closed at H2 until env-setup evidence passes and task card order is reconciled.

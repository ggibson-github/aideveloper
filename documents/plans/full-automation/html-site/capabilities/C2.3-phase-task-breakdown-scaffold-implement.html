phase task-breakdown scaffold implement

Task-breakdown and scaffold follow approved design: phased TODOS, task cards, environment-setup tasks ordered before package installs, then scaffold-project creates repo layout and CI skeleton. Missing setup tasks violate AGENTS.md rule 17—implement must not start without runnable environment.

Task cards carry verify commands and evidence paths for Plane G. Scaffold outputs are inputs to implement-feature loop; dual-write marks phase complete in journal/state.

How it works

  1. Step 1A goal slice is satisfied only when all tasks in scope are complete, tests pass or documented exceptions are recorded in the journal, C3.3 evidence exists on disk, and no blocking questions remain—distinct from a single task done or phase done.
  2. Step 2C3.4 rollup checks aggregate task completion and evidence paths for the slice boundary before A2.4 triggers goal_verify.
  3. Step 3A1.3 goal_verify must be ready to run—success criteria machine-checkable and evidence paths valid—before the conductor sets goal.state=verifying and requests H3 sign-off.
  4. Step 4journal-keeper dual-writes completion status at slice and goal levels separately in journal and STATUS.md so operators see partial progress versus achieved without conflating task, slice, and goal states.
  5. Step 5H3 is still required for verified goal acceptance unless pack policy waives it; if rollup fails, evidence is missing, or blocking questions persist, pursuit fails closed at H2 rather than marking the slice satisfied.

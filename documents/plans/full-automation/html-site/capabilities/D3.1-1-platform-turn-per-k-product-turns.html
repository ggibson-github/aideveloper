D3.1 · Platform evolution plane

1 platform turn per K product turns

platform turn per k product turns

Default scheduling grants one platform turn every K product turns—commonly K=5—encoded in platform.drain_policy.product_steps_per_platform_turn. Autopilot evaluates (steps_total % K) == 0 after each product phase completes; when true, the next iteration is platform dequeue instead of implement or design.

If the improvement queue is empty, the platform turn is skipped and the project continues—delivery does not sit idle waiting for self-improvement work. K is pack-configurable (D5.1) without forking schedulers. Overrides from D3.2, D3.4, and D3.5 layer on this baseline.

How it works

  1. Step 1When compose-first cannot extend a catalog entry without breaking consumers or pack isolation, the conductor records the divergence in B4.4 and opens a fork with provenance back to the source promotion or parent artifact.
  2. Step 2Platform workers allocate a new catalog id, append an INDEX row, and wire a staleness graph node. before marking the fork item done in the promotion queue.
  3. Step 3Downstream task cards migrate explicitly from the old id to the new entry; silent aliasing is forbidden so goal_verify evidence remains tied to one canonical component.
  4. Step 4Pack fragments from D4.6 may fork upward into the template-pack namespace when company boundaries require a separate reusable artifact.
  5. Step 5Fork without provenance fails closed at H2 and blocks SEC-14 gap-analysis trust rather than publishing an untraceable catalog entry.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Platform queue depth 0 but product blocked on missing playbook — D3.3 priority cut skips platform drain.
  • Edge case D3.1 variant 4: verify state dual-write before continuing pursuit.

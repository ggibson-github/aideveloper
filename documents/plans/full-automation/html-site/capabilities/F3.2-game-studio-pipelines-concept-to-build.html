game studio pipelines concept to build

Game studio pipelines walk concept → mesh → rig → animation → engine import → QA → build—each phase bound to roles and gates in pipelines/*.yaml. Phase order prevents animator work before rig exists and blocks build before QA evidence passes.

Pipeline phases reference pack playbooks (F1.6) and seed task cards (F1.7) for implement turns. Integration manifest declares which phases may parallelize (e.g. multiple asset lanes). goal_verify at ship uses F3.4 suite atop generic F1.8.

How it works

  1. Step 1At H1 plan approval for a game-studio goal, program-scoper binds the active pack's pipelines/*.yaml so pursuit phases follow concept → mesh → rig → animation → engine import → QA → build in enforced order.
  2. Step 2Each phase turn loads role-specific playbooks (F1.6) and seeded task cards (F1.7) so implement workers advance only the current pipeline gate—not downstream animator or build work.
  3. Step 3The integration manifest declares which asset phases may parallelize; the conductor schedules lane workstreams without violating ordering gates (e.g. no animator tasks before rig evidence passes).
  4. Step 4QA evidence must pass before build-phase tasks enqueue; partial phase completion rolls up in state but does not unlock ship until F3.4 atop F1.8 succeeds.
  5. Step 5If a worker attempts an out-of-order phase, required prior gate evidence is missing, or parallel lanes collide on the same asset lease, pursuit halts at H2 until the conductor reorders work at H1.

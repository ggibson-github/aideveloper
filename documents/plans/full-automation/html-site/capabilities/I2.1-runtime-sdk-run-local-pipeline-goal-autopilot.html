runtime SDK run-local-pipeline goal_autopilot

scripts/automation/run-local-pipeline.py is the SDK/local daemon entry for goal_autopilot—repeated continue steps without manual /continue, using the same journal/state contract as IDE pursuit (A3.4).

The daemon runs S0 check-pipeline-blocked before each iteration and stops on gates, budget exhaustion, or integrity failures. It spawns Cursor agents or SDK equivalents per model-policy; unattended 24/7 operation still respects H2/H3 stops. Pair with worker-runs audit (H5).

How it works

  1. Step 1When goal_autopilot is active, scripts/automation/run-local-pipeline.py wakes on schedule and runs the continue skill without an operator typing /continue, reading next_action from journal/state.json under the same contract as IDE pursuit (A3.4).
  2. Step 2Before each daemon iteration, S0 check-pipeline-blocked runs; gates, blocking questions, budget caps, or corrupt state stop the loop and surface H2 instead of advancing pursuit.
  3. Step 3Each iteration spawns Cursor agents or SDK equivalents per docs/operator/model-policy.json—genius-tier conductor for orchestration, economy workers for bounded tasks—while dual-writing journal and state after every step.
  4. Step 4When in-scope work is complete, the daemon routes to goal_verify rather than another implement task, then holds for H3 sign-off if verification passes.
  5. Step 5Worker run metadata feeds H5 audit; on integrity validation failure or repeated verify failure, autopilot halts with structured H2 and last_failure recorded in the journal.

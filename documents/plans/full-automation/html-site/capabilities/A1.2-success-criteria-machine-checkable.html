A1.2 · Pursuit & control plane

success criteria machine-checkable

success_criteria machine-checkable

A goal that cannot be checked mechanically is not a goal the system can pursue autonomously. This capability requires success_criteria that scripts or tests can evaluate—clear predicates, exit codes, or artifact checks—not subjective “looks good” language.

Success criteria are the bridge between intent (approved at H1) and goal verification. Each criterion should map to a command, file presence, or metric the verifier can run without asking a human. Vague criteria force H2 blockers and stall the always-on loop.

When criteria change mid-pursuit, the conductor must dual-write the update and re-run preflight so downstream steps do not assume an obsolete definition of done. Poor criteria are a common root cause of false “achieved” states; this spec exists to prevent that class of failure.

How it works

  1. Step 1At H1 plan approval, the conductor records success_criteria[] in state.goal as machine-checkable predicates—exit codes, artifact paths, or metric thresholds—that A1.3 and the verifier can run without human judgment.
  2. Step 2On each pursuit wake, A2.1 rejects criteria that cannot map to a script, test, or artifact check; subjective language forces an H2 blocker before any implement step proceeds.
  3. Step 3The conductor binds each criterion to a concrete verify command or evidence path in task cards so workers know which checks must pass before goal.state may advance past pursuing.
  4. Step 4If operators amend success_criteria mid-pursuit, the conductor dual-writes journal and state.json, then re-runs preflight so downstream steps do not pursue an obsolete definition of done.
  5. Step 5When verification cannot reconcile criteria against collected evidence, pursuit fails closed at H2 rather than transitioning to achieved or H3 pending.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A1.2 variant 4: verify state dual-write before continuing pursuit.

role tool permissions MCP CLI allowlist

Tool permissions per role encode MCP server allowlists, CLI patterns, and write scopes—programmer may run build CLI; designer may run Blender MCP; governance may read-only audit tools. Bindings live in roles/*.yaml and mirror operator policy overrides (B5.2).

Violations block spawn or fail verifier with permission error—never silent downgrade to broader tools. External game tools (F3.3) and data deploy CLIs declare here. Pack authors test permission matrix with conformance scripts before publish.

How it works

  1. Step 1On each worker spawn, orchestrate-subagents loads the active role's roles/*.yaml bindings and attaches MCP server allowlists, CLI patterns, and write scopes to the worker contract before any tool invocation.
  2. Step 2Tool-operator and MCP preflight checks each requested path and command against the role permission matrix; a pattern outside the allowlist blocks spawn or fails the verifier with a permission error—never a silent downgrade to broader tools.
  3. Step 3Domain packs extend the matrix for external tooling—Blender MCP for designer roles (F3.3), build and deploy CLIs for engineer roles—while B5.2 keeps product vs platform permission profiles aligned with pipeline_id.
  4. Step 4Pack authors run conformance scripts against the permission matrix before publish; pursuit with an active pack rejects worker runs that reference tools not declared in the role profile.
  5. Step 5Permission violations or allowlist drift between state.company.active_role and pack YAML stop pursuit at H2 with the blocked tool pattern and owning role—operators reconcile bindings at H1 before resume.

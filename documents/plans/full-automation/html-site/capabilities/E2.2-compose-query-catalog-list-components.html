compose query catalog list-components

Catalog query runs through S0 list-components.py (or successor) against the umbrella and child INDEX manifests—returning candidate paths, maturity tier, and component type. Queries are deterministic; conductors use output rather than re-deriving hits in chat.

Input is the resolved capability from E2.1; output feeds E2.3 ranking and task card Components (E2.4). Empty results trigger E2.5 L0 proceed plus promotion enqueue—not silent invention.

How it works

  1. Step 1At the start of each continue, autopilot, and run-local-pipeline iteration, A2.1 runs scripts/automation/check-pipeline-blocked.py with goal_autopilot semantics instead of inferring stop conditions in chat.
  2. Step 2The script encodes hard stops from A4: human gates (H1/H2/H3), budget exhaustion from A1.4, corrupt state., and verify regressions.
  3. Step 3A3.2 goal autopilot consults check-pipeline-blocked each wake.—conformance means autopilot cannot override a would-stop result.
  4. Step 4While gates_pending, blocking_questions, or hard-block stops remain, state preserves those fields until explicit operator approval or waiver—autopilot does not clear them by continuing turns.
  5. Step 5If check-pipeline-blocked would stop but pursuit advances anyway, preflight fails closed at H2 with the script output attached rather than stacking another pipeline step.

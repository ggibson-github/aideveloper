G2.3 · Verification & quality plane

goal verify blocks H3 until pass

goal_verify blocks H3 until pass

Human H3 sign-off stays blocked until goal.verify_command exits zero. Passing task-level evidence alone does not set hitl.pending=H3; goal verify success transitions goal.state through verifying and only then surfaces H3 for operator acceptance.

Automatic H3 clearance on implement completion is forbidden—operators sign verified outcomes, not assumed ones (INTRO-1.2). Failed goal verify packages structured H2 with aggregated log paths; pursuit resumes only after fix and re-run.

flowchart TD
  trigger[Trigger condition] --> pre[Preconditions S0]
  pre --> exec[Execute step logic]
  exec --> verify[Verify output]
  verify -->|pass| done[Mark queue item done]
  verify -->|fail| h2[H2 blocker]

How it works

  1. Step 1When scope completes, pursuit runs goal.verify_command. and sets goal.state to verifying before any H3 prompt is surfaced.
  2. Step 2Task-level last_verify: passed from G1.1 alone does not set hitl.pending=H3—goal verify exit zero is required.
  3. Step 3On goal verify pass, the conductor dual-writes verifying → H3 pending and waits for explicit operator sign-off.—automatic H3 clearance on implement completion is forbidden.
  4. Step 4Failed goal verify aggregates unit, integration, e2e, and tool logs. into structured H2 notes; pursuit resumes only after fix and re-run.
  5. Step 5If goal verify is bypassed, evidence is stale, or H3 is requested before verify passes, pursuit stops at H2 fail-closed rather than marking the goal achieved.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G2.3 variant 4: verify state dual-write before continuing pursuit.

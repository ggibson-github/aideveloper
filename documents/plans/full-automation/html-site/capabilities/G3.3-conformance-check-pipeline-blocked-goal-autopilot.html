conformance check-pipeline-blocked goal_autopilot

check-pipeline-blocked.py with goal_autopilot semantics defines when pursuit must stop: human gates (H1/H2/H3), budget exhaustion, corrupt state, and verify regressions. Autopilot and run-local-pipeline consult this script each iteration instead of inferring stop conditions in chat.

Conformance means autopilot cannot override a would-stop result—gates_pending and blocking_questions remain until explicit operator action. Pair with A2.6 and A4 stop taxonomy.

How it works

  1. Step 1scripts/headless-verify.py runs without an IDE session, reading journal/state.json and the active task card from disk to delegate verification to verify-router.py as an S0 check—no conductor LLM turn unless explicitly configured.
  2. Step 2Headless mode mirrors the IDE verifier contract: it executes file-derived test or tool commands from the task card and does not improvise scope beyond what state and cards declare.
  3. Step 3On exit 0, evidence logs land under evidence/ with the same immutability rules as the IDE verifier skill (H4), and last_verify is set to passed in state.json for the conductor to read on the next wake.
  4. Step 4Operators invoke headless-verify locally between IDE sessions, and CI jobs (I3.2) reuse the same entry point so implement batches get verified without consuming a genius-tier orchestration turn.
  5. Step 5On non-zero exit, headless-verify sets last_verify to failed in state.json; pursuit fail-closes. until the conductor or operator reruns verify with green evidence—CI failure alone does not clear H3 or design gates.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • Flaky test — escalation S4 once, then H2 with evidence log; no silent retry loop.
  • Edge case G3.3 variant 4: verify state dual-write before continuing pursuit.

A1.1 · Pursuit & control plane

goal id, parent goal, and goal type

goal_id parent_goal goal_type

Every pursuit in the expert system begins with a goal record that can be found again after restarts, nested under the right parent, and classified by type. This capability defines the three identity fields that make that possible: a unique goal_id, an optional parent_goal for tree structure, and goal_type (app, feature, milestone, or company_ops).

The type enum is not cosmetic. It tells preflight, routing, and verification which contracts apply—a milestone goal may require manifest approval while a feature goal binds to task cards and evidence paths. Nesting via parent_goal lets a program decompose into parallel streams without losing roll-up status at H3.

Operators set these fields at H1 when approving a plan; afterward the conductor and S0 scripts maintain them through dual-write to journal and state.json. If identity drifts (duplicate ids, wrong parent), pursuit turns attach evidence to the wrong scope and goal_verify cannot succeed. See Vision §3 — Branch A — Pursuit & control plane for how the goal model anchors Plane A.

How it works

  1. Step 1At H1 plan approval, the conductor stamps goal_id, optional parent_goal, and goal_type into state.json and the journal—the identity record every pursuit wake reloads before routing or verification.
  2. Step 2On each wake, A2.1 reads goal_type to bind contracts: milestone goals require manifest gates, feature goals bind task cards and evidence paths, and company_ops goals follow the ops queue.
  3. Step 3When parent_goal is set, child pursuits inherit roll-up scope so parallel program streams report status to the parent without duplicating identity at each leaf or losing H3 roll-up.
  4. Step 4The conductor and journal-keeper dual-write any H1 amendments to these fields; B1.1 scripts refuse progression when goal_id collides or parent_goal points to a missing ancestor.
  5. Step 5If identity drifts—duplicate ids, wrong parent—pursuit halts at H2, evidence must not attach to the wrong scope, and A1.3 cannot pass until fields are reconciled.

Edge cases

  • Operator closes laptop mid-loop — state.json must resume from last good dual-write.
  • Concurrent manual edit to queue JSON — conductor reloads queue each wake; last writer wins with journal note.
  • goal_verify passes but H3 rejected — goal.state returns to pursuing with rejection notes.
  • Edge case A1.1 variant 4: verify state dual-write before continuing pursuit.

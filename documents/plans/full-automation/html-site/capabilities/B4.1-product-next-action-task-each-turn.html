PRODUCT next action task each turn

PRODUCT next_action task each turn

Each product turn executes the current next_action task from state.json—one task card, one skill phase—aligned with Plane C pipelines. The conductor does not skip ahead to push or refactor while implement evidence is missing.

Task selection is S0 where scripts exist; otherwise conductor reads state and journal mirror. This capability binds Plane B routing to Plane C execution order so interleaved product and platform scheduling does not desynchronize project progress from recorded next_action.

How it works

  1. Step 1Each product turn executes exactly one next_action from state.json—one task card or one skill phase aligned with Plane C pipeline order—without skipping to push or refactor while implement evidence is missing.
  2. Step 2B1.1 sync-state and route-tier read state.json first when available; the conductor interprets the journal mirror only when S0 output is silent.
  3. Step 3Under interleaved B4.2 scheduling, product turns still advance next_action atomically so Plane C progress does not desync from recorded state.
  4. Step 4Workers execute the delegated slice of next_action; only the conductor advances past the task after verify passes or structured H2 is recorded.
  5. Step 5If journal and state.json disagree on next_action, or required evidence is missing, A2.1 blocks the turn until dual-write reconciles.

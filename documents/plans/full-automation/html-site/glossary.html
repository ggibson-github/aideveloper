Architecture

Capability

One clearly defined piece of what the automated system must be able to do — like a written contract that says what should happen, how to check it worked, and what can go wrong. The book lists 239 of these; together they describe the whole design.

Architecture

Branch

A chapter-sized grouping of related capabilities in this book — for example everything about verification, or everything about how the system runs day to day. Branches help you navigate a large design without reading it all at once.

Architecture

Architectural planes

Ten major chapters of the design (A through J), each owning one concern — pursuit, routing, product delivery, self-improvement, knowledge, organization, verification, persistence, runtime tools, and operator policy.

Architecture

Expert system

The automated software delivery system this book describes: it carries out projects with minimal human involvement, stops and asks for help when stuck, and keeps proof that each step actually worked before moving on.

Architecture

Evidence

Saved proof that a step really succeeded — for example a test log, a command output, or a file checksum. The system is not allowed to claim progress without evidence on disk that a computer can re-check later.

Architecture

Dual-write

Every important status change is saved in two places: a human-readable journal file and a machine-readable state file. If the IDE closes or the process restarts, the system can pick up where it left off without relying on chat memory.

Architecture

Catalog (compose-first)

The organization's indexed library of facts, playbooks, scripts, and reusable pieces. Before inventing something new, the system must search the catalog and reuse what already exists — like checking parts inventory before fabricating a new part.

Charts & metrics

Capability density by branch

A bar chart on the overview page showing how many capabilities are specified in each plane. Taller bars mean more design detail was written for that area. This measures documentation breadth, not how fast the software runs.

Charts & metrics

Capability density

How many capabilities belong to a given plane or topic — a simple count of how much of the design lives in that section.

Charts & metrics

Architectural scope by branch

A doughnut chart on the overview page showing each plane's share of all 239 capabilities — which parts of the design take up the most space on paper.

Charts & metrics

Architectural scope

How much of the overall design a plane or component owns, measured here by counting capabilities relative to the whole book.

Human touchpoints

H1 — initial planning gate

The first time a human must act: you describe the goal (a product, feature, or project) and approve the plan. After H1, the system is expected to work on its own until it needs help or finishes.

Full specification →

Human touchpoints

H2 — blocker assistance

A pause where the automation cannot continue without you — for example missing credentials, an unclear requirement, or a failed check it cannot fix alone. You provide what is missing; the system records the answer and resumes the same goal.

Full specification →

Human touchpoints

H3 — final sign-off

The last human step: after automated tests and checks pass, you formally accept or reject the completed work. Rejection sends the system back to fix issues rather than closing the goal.

Full specification →

Human touchpoints

Human-in-the-loop touchpoints

The rule that humans appear only at three moments: approve the plan (H1), unblock when stuck (H2), and sign off at the end (H3). Everything between those moments is meant to run without asking you to click Continue on every step.

Routing & tiers

S0 — deterministic scripts

The cheapest, most reliable kind of work: small programs that read files and run fixed steps with no AI guessing. The system must run these scripts first whenever they exist, before spending tokens on an AI model.

Full specification →

Routing & tiers

S1 — mechanical workers

Routine AI-assisted work with a tight brief — searching code, applying a task card, running a command — using a lower-cost model because the steps are well defined and low risk.

Full specification →

Routing & tiers

S2 — templated artifacts

Producing standard documents or artifacts from templates — design sections, task lists, diagrams — following patterns the organization already agreed on instead of inventing structure from scratch each time.

Full specification →

Routing & tiers

S3 — architecture / ambiguity

Judgment calls where there is no single script answer — trade-offs, merges after parallel work, or gaps in the written rules. A senior orchestration model handles these turns instead of a routine worker.

Full specification →

Routing & tiers

S4 — governance escalation

When the system must stop and package a clear question for a human because policy, legal, safety, or high-risk impact requires an explicit decision — not because a test failed.

Full specification →

Routing & tiers

Capability tiers S0 through S4

Five levels of how hard or expensive a turn of work should be: from fixed scripts (S0) up to governance escalation (S4). Routing picks the lowest tier that can safely do the job.

Routing & tiers

Conductor

The lead AI role that runs the show: it decides what happens next, assigns smaller worker AIs, merges their results, and writes the official record of progress. It does not do all the coding itself.

Operations & pursuit

Pursuit loop

The main repeating cycle after you approve a plan: check whether it is safe to act, do one step of work (product or platform), save progress, and repeat until the goal is done, you are needed (H2), or you sign off (H3).

Full specification →

Operations & pursuit

Product turn

One step of work directly on the current product, feature, or project — for example implementing a task, running tests, or updating design docs for that goal. This is the work the customer or team asked for.

Full specification →

Operations & pursuit

Platform turn

A step where the system works on itself instead of the current product: while delivering projects it notices repeated mistakes, manual work, or missing automation and queues improvements. On a platform turn it turns that queue into scripts, playbooks, or checks so the next project runs better. Product work and self-improvement alternate so delivery does not stop entirely for maintenance.

Full specification →

Operations & pursuit

Promotion queue

A to-do list of self-improvement items discovered during real project work — the same command run twice by hand, a pattern worth scripting, a playbook that should exist. Platform turns pull from this queue and mature each item into lasting automation.

Full specification →

Operations & pursuit

Goal verification suite

A final automated exam run when all tasks for a goal are finished. It aggregates tests and checks across the whole scope. Passing is required before the system asks you for H3 sign-off; failing sends work back for fixes.

Full specification →

Operations & pursuit

Autopilot

Mode where the system keeps taking pursuit steps on its own after H1 — without you typing Continue after every phase — until the goal is verified, blocked for your help, or finished.

# A5.3: default after H1 auto-enter goal_autopilot

**Parent:** [A5-index](A5-index.md) · **Branch A** · **Vision §3** · **Release:** v2.15

## Reader narrative
<!-- prose-source: agent plane-a 2026-06-28 -->

After H1 approves a plan, the default transition is automatic entry into goal autopilot—no manual “start pursuit” step. The goal-keeper skill (or equivalent conductor path) sets `autopilot.active`, populates goal fields from A1, and runs the first preflight in the same session when possible.

This default encodes “100% automation between touchpoints”: the operator supplies intent once, then observes until H2 or H3. Opt-out remains for strict HITL packs or debugging single steps. Pair with [A3.2](A3.2-goal-autopilot-until-goal-verify-or-hard-block.md) for the run mode entered here.

## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. When H1 plan approval clears `hitl.pending` and Resolved Q&A records the approval, goal-keeper sets `autopilot.active` and populates goal fields from [A1](A1-index.md) in the same dual-write turn—no separate manual start-pursuit command.
2. The conductor immediately runs [A2.1](A2.1-preflight-check-pipeline-blocked-extended.md) preflight and, if READY, enters [A3.2](A3.2-goal-autopilot-until-goal-verify-or-hard-block.md) goal autopilot in the same session when operator policy allows.
3. This default encodes 100% automation between touchpoints: intent is supplied once at H1, then pursuit runs until H2, verify stop, resource stop, or H3 per [A4](A4-index.md).
4. Strict HITL packs or debug single-step mode may opt out—`autopilot.active` stays false and [A5.1](A5.1-continue-resume-pursuit-if-not-blocked.md) drives one phase per Continue instead.
5. If goal fields are incomplete after H1, preflight fails closed at H2 rather than entering goal autopilot with ambiguous success_criteria or missing goal_verify_command.

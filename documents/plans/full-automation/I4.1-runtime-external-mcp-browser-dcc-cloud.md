## Behavior / step logic
<!-- timeline-source: agent cursor-agent 2026-06-28 -->

1. When spawning workers for implement or tool-operator tasks, the conductor loads the active MCP server list from template-pack role permissions ([B5.2](B5.2-role-to-pipeline-id-skills-tool-permissions.md)) and injects only approved browser, DCC, or cloud API tools into the spawn contract.
2. MCP tool calls execute inside the worker session as integration runtime—tool results return to the worker, but pursuit state transitions (`next_action`, `goal.state`, `evidence_files`) remain conductor-owned dual-writes to journal and state.json.
3. Browser MCP sessions follow lock/navigate/unlock discipline: the worker acquires browser lock before navigation, completes the scoped action from the task card, then releases—preventing parallel rabbit-hole automation across pursuit turns.
4. Auth failures, rate limits, or disallowed tool invocations abort the worker turn immediately; the conductor records structured H2 with credential pointers from docs/facts/INDEX rather than retrying with elevated permissions.
5. DCC and cloud MCP outputs land only on task-card-declared paths; the verifier attaches logs to evidence/ per [B2.4](B2.4-verifier-tool-operator-evidence.md) before pursuit marks the implement step complete.
